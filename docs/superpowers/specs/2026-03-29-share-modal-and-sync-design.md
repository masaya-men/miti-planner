# 共有モーダル ロゴ/画像修正 + 端末間同期 設計書

> 作成日: 2026-03-29 第39セッション

## 課題1: 共有モーダルのロゴ/画像修正

### 問題点

1. ロゴ切替（トグルON/OFF）が共有画像に反映されない（プレビューだけ変わる）
2. 削除→別画像ドロップで古い画像が小さいアイコンで出る
3. 変更ボタン→別画像ドロップで無限に生成中になる
4. 生成中でも共有ボタンが押せてしまう

### 設計方針

**既存のshareIdの共有データを上書き更新する方式**

- ロゴ追加/変更/削除/トグル切替のたびに `/api/share` を再呼び出し
- 新しいshareIdは作らず、同じshareIdのFirestoreドキュメントを上書き
- URLが変わらず、ゴミデータも増えない
- コスト影響: ほぼゼロ（ユーザーあたり数回の追加書き込み程度）

### UI変更

- 「変更ボタン」を廃止
- 操作は3つだけ:
  - **ドラッグ&ドロップ**: ロゴがない状態 → 追加。ある状態 → 置き換え
  - **追加ボタン**: ファイル選択ダイアログ。ロゴがない状態 → 追加。ある状態 → 置き換え
  - **削除ボタン**: ロゴを削除
- **共有画像ON/OFFトグル**: OFFならOGP画像なしでURLだけ共有
- **生成中ブロック**: 画像生成中は共有ボタン（URLコピー・X共有）を無効化

### 変更ファイル

| ファイル | 変更内容 |
|---------|---------|
| `src/components/ShareModal.tsx` | UI変更（変更ボタン廃止）、ロゴ操作後にshare再作成、生成中ブロック |
| `api/share/index.ts` | PUTメソッド追加（既存shareIdの上書き更新） |

### 動作フロー

```
モーダルを開く
  → POST /api/share（新規作成）→ shareId取得 → プレビュー表示

ロゴ追加/変更（ドラッグ&ドロップ or 追加ボタン）
  → Firebase Storageにアップロード
  → PUT /api/share（既存shareIdのロゴ更新）
  → プレビュー再読み込み
  → 完了まで共有ボタン無効化

ロゴ削除
  → Firebase Storageから削除
  → PUT /api/share（ロゴなしで上書き）
  → プレビュー再読み込み
  → 完了まで共有ボタン無効化

共有画像ON/OFFトグル
  → OFFの場合: OGP画像なしでURL共有（/api/share呼び直し不要、クライアント側の表示制御のみ）
  → ONの場合: OGP画像付きでURL共有
```

## 課題2: 端末間同期の信頼性

### 問題点

1. デバイスAで削除したプランが、デバイスBでリロードしても復活する
2. 削除済みプランが別端末で編集可能な状態が残る
3. localStorageとFirestoreのマージロジックが「localにあればFirestoreになくても残す」になっている

### 設計方針

**Firestoreを正（信頼できるデータ）として扱う + 保存時チェック**

Notionと同じアプローチ。リアルタイム同期（onSnapshot）は不要。

### 修正内容

#### A. ログイン時のマージロジック修正（planService.ts）

現在:
```
localにあってFirestoreにない → 残す（復活バグの原因）
```

修正後:
```
localにあってFirestoreにない:
  - ownerId === 'local'（未ログイン時に作った新規プラン）→ Firestoreにアップロード
  - ownerId === uid（以前ログイン中に作ったプラン）→ 別端末で削除されたとみなし削除
```

#### B. 保存時の存在チェック（usePlanStore.ts）

- `syncToFirestore`実行時、保存対象のプランがFirestoreに存在するか確認
- 存在しなければ:
  - 「この表は別の端末で削除されました」とトースト表示
  - ローカルからも削除して正しい状態にする

#### C. `_deletedPlanIds`の永続化は不要

- Firestoreを正とするため、メモリのみで問題なし

### 変更ファイル

| ファイル | 変更内容 |
|---------|---------|
| `src/lib/planService.ts` | `migrateLocalPlansToFirestore`のマージロジック修正 |
| `src/store/usePlanStore.ts` | `syncToFirestore`に存在チェック追加、削除通知 |
| `src/components/Layout.tsx` | マイグレーション処理の修正（必要に応じて） |

### コスト影響

- 追加のFirestore読み取り: ログイン時に1回（既存のfetchUserPlansで取得済みのデータを使うため追加コストなし）
- 保存時チェック: 保存するプランに対してgetDoc 1回（月数十回程度、無料枠に影響なし）
