# 探すページ 地図表示モード 設計書

> 2026-07-10 brainstorming でユーザー承認済み。公開前必須の大物3つの2つ目
> (原要望 = `docs/.private/2026-07-08-housing-release-feature-braindump.md` #4)。
> 実装は本設計書 → writing-plans の計画書を経てから。

## 1. 目的

**「自分が住んでいるエリアの近場にハウジングがあるか見て回れる」楽しさ**を作る。
探すの中央パネルを 一覧 ⇔ 地図 で切り替え、区画マップの上に登録ハウジングのカードが浮かぶ。
一覧の人気順・注目順に依存しない「場所からの発見」の入口 (ハウジンガーPF の「人気偏重を避ける」思想と同軸)。

## 2. 確定済みの設計判断 (2026-07-10 ユーザー確定・変更不可)

1. 地図の単位 = **住宅街 × 区** の区画マップ (ツアーと同じ地図素材)。
2. 登録がある番地には**吹き出し付きミニカードが常時浮かぶ** (「ぼんやり光るだけ」は不採用)。
3. ミニカードは hover で**拡大して探すのカードと同一部品**になる — 画像がよく見え、
   クリックで詳細・ツアーに追加・お気に入りが**探すと同じ操作**でできる。
4. ワールド選択ゲート = **A案**: 地図ボタンはいつでも押せる。未絞り込みなら地図モード内で
   「あなたのワールドは?」を案内 (押せないボタンにしない)。
5. 同一スポット複数件 = **代表1枚 + 件数バッジ (×N)**。実態はアパート / FCハウス+個室 /
   重複登録(ミス) の3パターン (ユーザー知見)。ホバー拡大時にその場所の家を切り替えられる。
6. 本街 ⇔ 拡張街 (番地1-30 / 31-60 で地図が2枚) = **切替タブ**。反対側の件数をタブに表示して見落とし防止。
7. 実装方式 = **探す専用の地図部品を新設 (案2)**。ツアーの地図 (`TourNavMap` 等) には**一切触らない**
   (共有部品の構造変更が実機で壊れた過去の教訓 [[feedback_structural_refactor_runtime_audit]])。
   地図画像・座標データの読み込み口だけ流用する。

## 3. 画面仕様

### 3.1 切替タブ

- 探す中央パネル上部に「一覧 | 地図」切替。`ListingGrid.tsx:14-16` に温存されていた
  ビュー切替の設計位置を使う (「ルート」は本設計のスコープ外)。
- 表示モードはセッション記憶 (`useHousingViewStore` の旧 `viewMode: 'map'|'pinterest'` は
  旧 Phase 2B の意味 (偽データ地図) なので**そのまま使わず**、`browseView: 'list' | 'map'` を新設。
  旧フィールドと到達不能な旧分岐 (`viewMode==='map' ? MOCK_LISTINGS : …`) は、地図実装で触る
  ファイルの範囲内でのみ整理する — 無関係な削除リファクタはしない)。

### 3.2 ワールド選択ゲート

- 地図モードに入った時点で `servers.length !== 1` なら、地図の代わりに選択画面:
  「あなたのワールドは?」→ DC → ワールドの2段選択 (既存フィルタと同じ選択肢データ)。
- 選んだ結果は **`useHousingFilterStore` に反映** (dc + servers=[選択1つ])。一覧に戻っても保たれる。
- 既に1つに絞れていれば即地図。

### 3.3 地図モードの操作列 (地図の上に1列)

- **住宅街切替** (ミスト/ラベンダーベッド/ゴブレットビュート/シロガネ/エンピレアム)。
- **区ドロップダウン** (1〜30区) + 前へ/次へ矢印。**各項目に件数を添える** (例「3区 (5件)」)。
- **本街/拡張街タブ** (件数付き。例「拡張街 (3)」)。
- 件数はすべて「現在のフィルタ適用後」の件数 (一覧と地図で数字が食い違わないこと)。

### 3.4 初期表示とゼロ件

- 初期表示 = 選択ワールドで**登録件数が最多の住宅街×区** (賑わいが最初に見える)。
- ワールド内 0 件 = 「このワールドにはまだ登録がありません」の案内 (housing トンマナの
  静かな空状態。`EmptyResult` の形を踏襲) + 一覧へ戻るボタン。

## 4. 地図上のカード仕様

### 4.1 ミニカード (常時表示)

- 登録がある番地 (および区のアパートスポット) の座標に、**吹き出し(しっぽ)付きミニカード**を常時表示。
  中身 = 代表画像サムネ (探すカードと同じ代表画像決定ロジック) + 番地ラベル。複数件は右肩に「×N」。
- 代表 1 件の選定 = その場所で `lastConfirmedAt` が最新のもの (「新しい確認 = 正」の既存方針)。

### 4.2 ホバー / タップで拡大

- hover (タッチ環境ではタップ) でその場で拡大し、**探すの `ListingCard` そのもの**を表示
  (画像スライドショー・ツアーに追加・お気に入り等、カードが持つ機能はそのまま)。
- 拡大は**同時に1枚だけ** (排他)。別のミニカードに hover / 地図の空白クリック / Esc で閉じる。
- 複数件スポットは拡大カード内に「1/3」の切替 (前後ボタン) を付け、その場所の家を順に見られる。
- 拡大カードは他のミニカードより前面。地図の端では画面内に収まる向きに吹き出しを反転。

### 4.3 パン / ズームとカードの大きさ

- 地図はパン / ホイールズーム / ピンチ対応 (新部品側で新規実装。ツアーの実装は参考に読むが import しない)。
- カードは**地図の座標に追従して動くが、画面上の大きさは一定** (ズームしても文字・画像が
  潰れない、地図アプリのピン方式)。

### 4.4 ツアーへの接続

- 拡大カードの「ツアーに追加」は一覧と同じくトレイ (右カラムの `TourTray`) に積まれる。
  地図モード中もトレイは見える位置に維持 (中央パネルだけが切り替わる)。

## 5. 技術構成

### 5.1 新設コンポーネント (ツアー無傷)

- `BrowseMapView` — 地図モードのコンテナ (ゲート判定 / 操作列 / 空状態)。
- `BrowseWardMap` — SVG 差し込み + パン/ズーム + カードレイヤ配置 (画面座標変換)。
- `MapSpotCard` — ミニカード⇔拡大カードの切替。拡大時は既存 `ListingCard` を描画。
- 集計純関数 `groupListingsByMapSpot(listings, area, mapKind)` — フィルタ済み listing 群を
  「番地スポット / アパートスポット」に集約し、代表・件数・座標参照を返す。
  `countListingsByWard(listings, area)` — 操作列の件数バッジ用。

### 5.2 流用する資産 (読み込みのみ)

- 地図素材: `WARD_MAP_LOADERS` / `useWardMapAsset` (`src/data/housing/wardMapManifest.ts` /
  `src/lib/housing/useWardMapAsset.ts`)。5住宅街 × 本街/拡張街 = 10 枚、座標 json 完備。
- 番地座標: 生成 json の `houses` 配列 (`plot` 30件 + `apart` 1件/枚)。アパートは**建物1点**の粒度
  (部屋単位の座標は存在しない → 集約表示と整合)。
- 代表画像: `ListingCard` 内ローカルの `representativeImage()` を共有関数に**切り出して**両者で使う
  (挙動は変えない。切り出しのみ)。

### 5.3 データフローとパフォーマンス

- listing は既に全件クライアント store (`useHousingListingsStore`) にあるため、地図の集計・件数は
  クライアント計算のみ (追加通信ゼロ・Firebase コスト増ゼロ)。
- 一覧と同じ `mergeListingsForViewer → applyFilters` の結果を地図にも流す (フィルタ完全共有)。
  **注意**: サイズフィルタは `size===undefined` のアパートを除外する既存仕様 (`applyFilters.ts:20-21`)。
  地図もこの結果に従う = 一覧と地図の件数は常に一致 (地図側で独自の緩和をしない)。
- スポット数は最大でも 31/枚 (番地30 + アパート1) なので描画負荷は軽い。パン/ズーム中の座標変換は
  transform ベースで行い、`scrollWidth` 等のレイアウト読み取りをイベント中にしない
  ([[reference_perf_forced_reflow_resizeobserver]])。拡大カード (ListingCard) は hover 時にだけマウント。

### 5.4 スマホ / アクセシビリティ

- v1 は PC 優先。タッチのパン/ピンチ/タップ拡大は動く状態にするが、スマホ専用レイアウト最適化は
  「探すページ全体のスマホ最適化」タスク (既存 backlog) に合流。
- ミニカードは `button` 要素 (キーボードフォーカス可・Enter で拡大)。拡大カードは Esc で閉じる。
- 実機確認は DPR 2.58 で行う (ミニカードのサイズ感・吹き出しの視認性)。

### 5.5 エラー時の挙動

- 地図素材の読み込み失敗 → 「地図を読み込めませんでした」+ 一覧へ戻るボタン (パネルは壊さない)。
- 座標 json に無い番地 (データ異常) のカードは描画スキップ (console.warn のみ・クラッシュしない)。

## 6. トンマナ・i18n

- `.claude/rules/housing-design.md` 準拠。吹き出しミニカード・件数バッジは**機能要素**なので可、
  装飾 999px ピル / 色付き alert 箱 / 過剰 glow は禁止。既存の housing トークンで配色。
- 文言は ja/en/ko/zh 4言語 parity。新規キーは `housing.map.*`。「ハウジング」統一。

## 7. テスト方針

1. `groupListingsByMapSpot` / `countListingsByWard` — 集約・代表選定 (lastConfirmedAt 最新)・
   アパート/FC個室/重複の集約・フィルタ済み入力前提、の純関数テスト
2. ゲート判定 (servers 0件/2件 → 選択画面、1件 → 地図) — コンポーネントテスト
3. 拡大の排他 (2枚目 hover で1枚目が閉じる)・Esc で閉じる — コンポーネントテスト
4. 本街/拡張街タブと区ドロップダウンの件数表示 — コンポーネントテスト
5. 実機: パン/ズーム追従・吹き出し反転・DPR 2.58 での視認性 (jsdom 不可のため目視)

## 8. スコープ外 (YAGNI・明示的に作らない)

- 「ルート」ビュー (一覧|地図|ルート の3つ目) / ワールド横断・全区俯瞰マップ / ヒートマップ表現
- スマホ専用レイアウト最適化 (既存タスクに合流)
- 地図からの新規登録導線 / 空き区画の表示 (登録データがある場所だけが主役)
- ツアー地図 (`TourNavMap`) の改修・共通フック化 (将来の任意タスク)

## 9. 実装順の目安 (計画書化の入力)

1. 集計純関数 + 代表画像の共有関数切り出し (テスト先行)

6. i18n 4言語 + CSS + 全体検証
