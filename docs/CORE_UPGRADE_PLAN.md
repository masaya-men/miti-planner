# Grapl プロジェクト：コアロジック・アップグレードおよび動的共有機能実装指示書

## 1. 依頼の目的
現在開発中のFF14軽減プランナー「Grapl（Miti Planner）」において、以下の2つの高度な機能を実装・統合します。
1.  **オートプランナーの刷新**: 従来の「降順貪欲法」から「焼きなまし法（SA法）」による全体最適化エンジンへのアップグレード。
2.  **動的OGP共有システム**: ユーザーロゴ（チームロゴ）とアプリロゴを合成したOGP画像を、Vercel Edge Functionでオンデマンド生成する仕組みの構築。

既存の `GRAPL_PROJECT_PLAN.md` および技術スタック（React 19, Zustand, Firebase, Vercel）を前提とした実装を行ってください。

---

## 2. 【機能A】次世代最適化エンジン（SA版 autoPlanner.ts）

### 2.1 アルゴリズム概要
`src/utils/autoPlanner.ts` のロジックを、タイムライン全体の「生存率」と「人間らしさ」を最大化する「焼きなまし法」へ書き換えてください。

### 2.2 グループパッケージ運用の強制
「致死ダメージを1人分だけ下回ったからといって、グループ内の1人だけ軽減を外す」という挙動を禁止し、以下の「パッケージ単位」で配置を試行してください。
* **Route A (MT組)**: MT(リプラ/全体軽減), H1(60s軽減), D1(牽制), D3(レンジ軽減)を一括配置。
* **Route B (ST組)**: ST(リプラ/全体軽減), H2(陣・セットスキル), D2(牽制), D4(アドル)を一括配置。

### 2.3 スコアリング・ルール
以下の優先順位でスコア（点数）を算出し、数千〜1万回のシミュレーションで最高得点のプランを採用してください。
1.  **絶対生存（最優先）**: 全イベントで被ダメージ < 100% の場合、巨大なボーナス。
2.  **リキャスト遵守**: リキャスト違反がある場合は大幅減点。
3.  **パッケージ運用**: 指定されたグループセットが欠けずに配置されている場合に加点。
4.  **戦略的無敵**: バフが枯渇するTBに対し、正しく Pattern 4（無敵）を割り当てている場合に加点。

## 3. 【機能B】動的OGP生成 & ユーザーロゴ管理システム

### 3.1 ユーザーロゴの永続化（Firebase Storage）
* **要件**: 1ユーザーにつき1枚のみロゴを許可。アップロード時に `users/{userId}/team-logo.webp` として保存（既存画像は上書き）。
* **最適化**: アップロード前にフロントエンド（Canvas API）で 400x400px 程度の WebP 形式にリサイズ・圧縮し、ストレージ容量を節約すること。

### 3.2 オンデマンドOGP生成（Vercel Satori / Edge Function）
* **生成フロー**: 共有URL（`/share/[planId]`）がSNSに貼られた際、API（`/api/og`）が呼び出される。
* **合成内容**:
    * 背景：プランに応じた背景（コンテンツ名を表示）。
    * 左上：ユーザーが設定した「チームロゴ」（未設定時は非表示）。
    * 右下：「Grapl」のアプリロゴ（固定）。
    * 中央：プランのタイトル、ボス名、および代表的な軽減配置のプレビュー。
* **データ消費**: 画像ファイルとしては保存せず、リクエストごとに生成してSNS側のキャッシュを利用すること。

## 4. UI/UXへの統合

### 4.1 共有確認モーダル（ShareModal.tsx）
共有ボタン押下時に以下の機能を持つモーダルを表示してください。
1.  **OGPプレビュー**: 現在設定されているロゴが含まれた画像プレビュー。
2.  **ロゴ設定トグル**: 「チームロゴを表示する/しない」の切り替え。
3.  **ロゴ貼り替え**: その場で新しい画像をアップロード（上書き）し、プレビューに即時反映。

## 5. 実装時の注意点（エンジニアリング要件）
* **ステート管理**: ログイン状態やロゴURLは `useAuthStore.ts`、軽減プランデータは `useMitigationStore.ts` と連携すること。
* **型定義**: `src/types/index.ts` に必要な型（`logoUrl` など）を追加すること。
* **パフォーマンス**: SA法の計算は Web Worker を使用するか、メインスレッドをブロックしない程度に分割実行することを検討してください。
