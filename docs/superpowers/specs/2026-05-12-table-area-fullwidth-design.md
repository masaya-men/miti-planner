# 表エリア全幅化 / メンバー列幅拡張 設計書

最終更新: 2026-05-12 セッション 17

## 背景

セッション 14 で **sizing 思想 v2** (container max-width 1489) を導入し、 「開発者画面 1489 でピクセルパーフェクト、 大画面は余白拡大」 という設計方針を全プロジェクトに適用した。 本来の目的の 1 つは「フォーカスモード時にタイムラインがユーザー画面の右端まで広がる」 ことだったが、 メンバー列幅 (T/H 126px / DPS 53px) の合計が利用可能幅に届かず、 タイムライン右側に約 170-200 CSS px の空白が残っている。

セッション 16 では「126/53 対称化」 を完了し、 罫線中央基準の対称性とサブピクセル累積補正を確立した。 本設計はその対称性思想を継承しつつ、 列幅を 1489 にピッタリ収まる値に再定義することで、 「画面全体に描画される」 視覚的な美しさを実現する。

## 課題 (実測)

ユーザー画面 (CSS viewport 1489px、 DPR 2.58、 フォーカスモード時) の実測値:

| 項目 | 値 |
|---|---|
| ユーザー画面 CSS 幅 | 1489px |
| 右端フォーカスボタンパネル (`Layout.tsx:642-697`) | 24px |
| Windows Chrome スクロールバー | 約 17px |
| **利用可能幅** | **約 1448px** |
| 現状タイムライン総幅 | 約 1295px |
| 他列合計 (フェーズ + ラベル + 時間 + 敵の攻撃 + 元ダメ + 軽減後) | 約 579px |
| 現状メンバー列合計 (T/H 126×4 + DPS 53×4) | 716px |
| **右側の空白** | **約 153px** |

フルパーティ (T2/H2/DPS4) 編成でも右側が埋まらないため、 ユーザーは「画面全体に描画されていない」 と感じる。

## ゴール

ユーザー画面 1489 でフォーカスモード時に、 タイムラインの右端がフォーカスボタンパネル直前までピッタリ到達する。 通常モードでは意図的に溢れさせて「サイドバー閉じる」 導線として機能させる (= UX 上の積極的設計)。

## 確定事項

| 項目 | 値 | 補足 |
|---|---|---|
| 既存フォーカスモード機能 | そのまま | F キー動作、 ヘッダー折りたたみ、 サイドバー閉鎖は触らない |
| 基準画面幅 | 1489 CSS px | sizing 思想 v2 維持 |
| T/H 列 アイコン配置エリア | 126 → **151px** | 5 → **6 アイコン対称** (セッション 16 の対称性ロジック踏襲) |
| DPS 列 アイコン配置エリア | **53px** 維持 | 2 アイコン対称、 変更なし |
| 各メンバー列の左右マージン | **約 3.3px** ずつ (初期値) | 8 列で計 52.8px、 残空白を吸収。 ローカル試作で 3.0〜3.5px の範囲で微調整可 |
| 適用範囲 | **全モード共通** | 通常モード / フォーカスモードで同一値 |
| 通常モード時の振る舞い | **あえて溢れさせる** | 横スクロール発生 → サイドバー閉じる導線として活用 |
| 大画面 (1920+) 対応 | container 中央寄せ + 余白拡大 | sizing 思想 v2 通り、 列幅は固定 |
| アイコンサイズ | 現状維持 | T/H 列で 5 → 6 アイコン並ぶが、 1 アイコン約 22-25px の密度は変えない |

## 計算根拠

```
[現状]
他列合計                          579px
T/H 列 (126 × 4)                  504px
DPS 列 (53 × 4)                   212px
─────────────────────────────────
コンテンツ合計                   1295px
利用可能幅                       1448px
─────────────────────────────────
右側空白                          153px

[拡張後]
他列合計                          579px  (変更なし)
T/H 列 (151 × 4)                  604px  (+100)
DPS 列 (53 × 4)                   212px  (変更なし)
メンバー列マージン (3.3 × 2 × 8)   52.8px (+53)
─────────────────────────────────
コンテンツ合計                  1447.8px ≈ 1448px (ピッタリ)
```

## 採用設計 (実装アプローチ A: CSS 変数分離)

セッション 16 の「対称値とサブピクセル補正は CSS 変数で一元管理」 という思想を踏襲。 マージンも独立した変数として持ち、 後の微調整 (ローカル試作時の罫線位置確認、 将来の調整) を容易にする。

### CSS 変数定義 (`src/index.css`)

```css
:root {
  /* 既存 (セッション 16) */
  --col-th-w: 151px;            /* 6 アイコン対称、 126 から拡張 */
  --col-dps-w: 53px;            /* 2 アイコン対称、 維持 */

  /* 新規 */
  --col-member-pad-x: 3.3px;    /* 各メンバー列の左右マージン (初期値、 ローカル試作で 3.0〜3.5px の範囲で微調整) */
}
```

なお、 上記の利用可能幅 1448px は計算値であり、 実機ではブラウザ実装やフォント拡縮等で数 px の揺れがあり得る。 ローカル試作時にユーザー画面で実測し、 マージン値を最終確定する。

### グリッドテンプレート計算

各メンバー列の総幅を `calc(var(--col-th-w) + var(--col-member-pad-x) * 2)` (T/H 用) と `calc(var(--col-dps-w) + var(--col-member-pad-x) * 2)` (DPS 用) で表現する。 グリッドテンプレートカラム定義箇所を全て更新。

### マージンの実装方法

各メンバー列の `<div>` (またはセル) に `padding-left: var(--col-member-pad-x); padding-right: var(--col-member-pad-x);` を適用。 内側の「アイコン配置エリア」 は従来通り 151px / 53px の中で対称配置される。

罫線 (= 列の境界線) は**メンバー列の外側** (= マージンの外) に描画されるため、 セッション 16 で確立した「罫線中央基準の対称性」 はマージン追加後も維持される。

## 影響範囲・確認項目 (ローカル試作で目視チェック)

ローカル試作ブランチで以下を全て確認してから本実装に進む:

1. **罫線位置** — セッション 16 の対称性 (中央基準) が維持されているか、 0.5px ズレが再発しないか
2. **ジョブアイコン (列ヘッダー)** — 各メンバー列の上部に表示されるジョブアイコンのサイズ・位置が崩れないか
3. **グリッドライン** (垂直罫線) — 各メンバー列の左右マージン部分にどう描画されるか、 不自然な「太い線」 が出ないか
4. **配置済み軽減アイコン** — 既存プランのアイコンが正位置に配置されるか (x 座標計算ロジックの整合性)
5. **メンバーヘッダー文字** (T1, T2, H1...) — 列幅変更による文字位置の変化
6. **空行非表示モード** (hideEmptyRows) — 列幅変更で動作するか
7. **モバイル / タブレット** — md 未満は別レイアウト、 影響なしを確認
8. **既存機能** — 軽減追加 / 削除 / ドラッグ / シェア / オートプラン / FFLogs インポート / モーダル系が動作するか
9. **左飛びバグ** (セッション 16 末で治療) — 列幅変更で再発しないか
10. **0.5px 対称ズレ** (セッション 16 末で未完了の積み残し) — 列幅広がる分、 相対的に目立たなくなる可能性、 同時治療の機会

## 進め方

1. **spec レビュー** ← 今ここ。 ユーザー承認待ち
2. **ローカル試作ブランチ作成** `feat/table-area-fullwidth`
3. CSS 変数だけ書き換えて、 ユーザー画面 (CSS 1489) で目視確認
4. 上記「影響範囲・確認項目」 を全て確認
5. 視覚的・機能的に OK なら writing-plans でフルプラン作成
6. plan に沿って実装 → ローカル動作確認 → push → 本番デプロイ → 本番確認
7. main にマージ後、 後続タスク (リキャスト専用行 / 効果中スキル最上行残し) に進む

## スコープ外 (やらないこと)

- フォーカスモード機能自体の変更 (`F` キー動作、 トグル UI など)
- container max-width 1489 の変更
- 通常モード時の表示調整 (= あえて溢れさせる UX、 これは意図的)
- 後続タスク 2「リキャスト専用行」 の実装 → 別 spec で扱う
- 後続タスク 3「効果中スキル最上行残し」 の実装 → 別 spec で扱う
- 1920+ ユーザー向けの特別対応 (= sizing 思想 v2 通り、 余白拡大維持)
- アイコンサイズ自体の変更 (= 6 アイコン対称になるが 1 アイコンの大きさは変えない)

## デザインの哲学

> 通常モードはあえて溢れさせて「サイドバー閉じれば見やすくなる」 ことに気付かせる導線にする。
> フォーカスモードはご褒美的にピッタリ収まる体験。

これは sizing 思想 v2 (ユーザー画面 = ピクセルパーフェクト) を強化する位置付け。 「制約を見せて、 解決手段に誘導する」 UX パターンとして機能する。
