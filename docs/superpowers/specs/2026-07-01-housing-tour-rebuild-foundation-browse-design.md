## 1. プログラム全体像 (参考・詳細設計は各スパンで)

参考UIは **上部タブで切り替わる7ページ + 共通シェル** のアプリ。

| # | ページ | URL(予定) | 中身 | 挙動の既存度 |
|---|---|---|---|---|
| 1 | 探す (Browse) | `/housing` | フィルター + 物件グリッド + ツアートレイ | 高 |
| 2 | ハウジング詳細 | `/housing/listing/:id` | ギャラリー + 情報 + 周辺ミニマップ + 報告 | 中(既存ページ有) |
| 3 | お気に入り | `/housing/favorites` | 複数選択グリッド + ツアートレイ + はじめての方へ | 中(モーダル実装済) |
| 4 | ツアーを組む | `/housing/plan` | 4ステップ(候補→順番→最適化→確認) + 効率スコア | 低(最適化は新規) |
| 5 | ツアー中 | `/housing/tour/:tourId?` | 進捗リング + LIVE地図 + 次の目的地 + 報告 | 低(地図配線が核) |
| 6 | 登録する | `/housing/register` | 6ステップフォーム + 入力/重複チェック + 土地プレビュー | 中(登録実装済) |
| 7 | マイページ | `/housing/mypage` | 統計カード + 登録ハウス表 + 報告対応 + 保存ツアー + お気に入り | 低(集計は新規) |

**進め方 = 土台を先に作り、以後1ページずつ「見た目を完璧に→実挙動を配線」で仕上げる**。各ページは独自の spec→plan→build を持つ。完成ページは死んだモックでなく実際に使える状態にする。

### 前提の確定事項 (全スパン共通)
- **ツアーは未公開**（導線ボタン無効化済）= 自由に作り直して OK。
- **地図表現 = B案**（ツアー全物件を1枚のエリア地図に配置・線で連結）。**歩けない境界では移動案内**（同区内=徒歩／別区=区移動／別ワールド=ワールド移動／別DC=DC移動）。※ナビ実装はツアー中スパン。
- **タブ = URLベース**（各ページに実ルート。ブックマーク/共有/戻る/リロードが効く）。
- **広告 = 最小の予約枠のみ**（`AdSlot`。将来の広告収入余地を確保、今は "Sponsored" プレースホルダ）。
- **体感速度重視**（perf を最初から設計に織り込む）。
- **登録UX**: 「何がどう足りないか」を具体的アクションで示す（最初から✅を付けない。[[feedback_form_ux_progress]]）。※登録スパン。
- **生きたカード = Allmarks流用**（動画プレビュー/spotlight再生/perf技）。

## 2. 第1スパンのスコープ

**作るもの = 土台（シェル + 共通デザインキット）+ 探すページ1枚を完璧に。**

土台だけでは中身が空で確認しづらいため、**土台の実証として「探す」を本気で1枚仕上げる**。ここで作る共通部品が残り6ページの材料になる。

### スコープ外 (このスパンではやらない)
- お気に入り/ツアーを組む/ツアー中/登録/マイページ の各ページ本体（タブは置くが中身は次スパン以降。暫定で既存実装 or プレースホルダに接続）。
- 地図のツアー配線・移動案内・ルート最適化・効率スコア（ツアー中/組むスパン）。
- 生きたカードの**動画再生**の作り込み（第1スパンは静止代表画像 + ホバー演出まで。動画spotlightは perf 検証を伴うので探す仕上げの後半 or 次スパンで段階導入）。※下記 §6 で段階を明記。

## 3. IA & ルーティング

現状 = `/housing` 1ワークスペース（左フィルター/中央/右）+ お気に入り/登録はモーダル。
→ **上部タブ6つ + 各ページに実ルート**へ再構成。

### ルート設計 (`App.tsx`)
- 既存 `/housing`(Workspace) を **`HousingShell`（共通レイアウト = ヘッダー+タブ+ステータス+`<Outlet/>`）** に載せ替え、子ルートで各ページを出す。
- `/housing` = 探す（index ページ）。他タブは §1 表のURL。
- `/housing/listing/:id`（既存詳細ページ）は**タブではなく**カードから遷移。シェルの外 or 中かは詳細スパンで確定（第1スパンでは既存挙動を壊さない）。
- 旧 `/housing/p/:listingId` `/housing/legacy` は当面残す（撤去は移行完了後）。
- **状態の持続**: タブ移動でフィルター/お気に入り/ツアートレイの store は保持（zustand なのでページ跨ぎで生存）。

### シェル構成
```
HousingShell
├─ SceneryVideo (背景・既存)
├─ AppHeader   … ロゴ / グローバル検索 / TabBar / テーマトグル / 通知 / アバター
├─ <Outlet/>   … 各ページ（探す・お気に入り・…）
└─ StatusBar   … BUILD版 / テーマ / 進行状況 / 言語 / (ツアー中は進捗)
```

## 4. 共通デザインシステム / 部品キット

**方針**: 質感は既存継承（`SceneryVideo` + `LiquidGlassPanel` の本物ガラス）。色/寸法/影/文字は**全て `--housing-*` トークン経由**（ハードコード禁止・`.claude/rules/housing-design.md`）。GPT画像で新たに要る色（ステップ番号の水色/緑、効率スコアの金星 等）は housing.css にトークン追加してから使う。

### 部品一覧 (第1スパンで新規 or 昇格)
| 部品 | 役割 | 由来 |
|---|---|---|
| `GlassPanel` | 全ページのパネルの器 | 既存 `LiquidGlassPanel` を汎用化 |
| `TabBar` | 上部6タブ + アクティブ下線 + 通知数バッジ | 新規 |
| `AppHeader` | ロゴ/検索/タブ/テーマ/通知/アバター | 既存 `TopBar` を再構成 |
| `StatusBar` | 下部バー | 既存を継承 |
| `ListingCard` | グリッドの1枚（探す/お気に入り/マイページ共通・生きたカード） | 既存カード + Allmarks化 |

## 5. 探すページ レイアウト

GPT画像1枚目（探す）準拠、ただし**GPT生成ゆえの無駄は削ぐ**（意味のないUI重複/冗長スペース/過剰広告枠を整理）。

```
┌ AppHeader (タブ: 探す[active]) ────────────────────────────┐
├ 左: フィルター  │ 中央: 物件グリッド        │ 右: ツアートレイ ┤
│  DC/ワールド/   │  [一覧|マップ|ルート]切替  │  n/20件         │
│  エリア/サイズ/ │  並び替え                  │  推定時間        │
│  テーマ/公開/   │  ListingCard × グリッド    │  ルート最適化(次) │
│  詳細条件       │  ページャ                  │  1..n 並べ替え可 │
│  [AdSlot 最小]  │                            │  この内容で開始   │
│                 │                            │  お気に入り一覧   │
│                 │                            │  [AdSlot 最小]   │
└ StatusBar ───────────────────────────────────────────────┘
```

- **中央の [一覧|マップ|ルート] 切替**: 第1スパンは「一覧」を完全実装。「マップ」「ルート」はタブは出すが**次スパン**（地図配線が要るため）。
- **データ**: `useHousingListingsStore`（実Firestore・既存）。フィルターは `applyFilters`（既存）。お気に入り/ツアーは既存 store 継承。
- **既存挙動の配線**: カードの♡=お気に入り toggle、「ツアーに追加」=`TourTray` へ追加（既存 `expandTourWithDuplicates` §3.8 の重複自動追加も継承）。
- **「この内容で開始」の第1スパンでの着地**: `setTourListings→startTour` でツアーを組成し `/housing/tour/:tourId` に遷移するところまで。**ナビ画面本体（進捗リング/LIVE地図/移動案内）はツアー中スパン**。第1スパンでは既存の最小ツアー表示 or 「準備中」プレースホルダに着地して構わない（組成が正しく走ることを確認できれば十分）。
- **空/読込/エラー**: 既存 `EmptyResult` / loading / error 表示を継承。

## 6. 生きたカード (Allmarks流用) — 段階導入

[[reference_allmarks_mycollage]] [[reference_allmarks_3frame_spec]] [[reference_perf_content_visibility]] [[reference_perf_forced_reflow_resizeobserver]] を流用。多数カードを軽く動かす。

- **段階1 (第1スパン)**: 静止代表画像 + `content-visibility:auto`/`contain-intrinsic-size` で描画スキップ。ホバーで軽い演出（拡大/縁の灯火）。既存 `HousingPlaybackContext`(`useHousingCardPlayback`) を土台に維持。
- **段階2 (探す仕上げ後半 or 次スパン)**: 動画カードの**画面内 spotlight 再生**（一度に1件だけ本再生、他は代表画像 or 3フレーム）。Allmarks 正規仕様=「3フレーム抽出 + 1件spotlight再生」の組合せ（[[reference_allmarks_3frame_spec]]、TG1でextract撤去=3フレーム不要ではない点に注意）。perf 実機検証を伴うため段階分離。

## 8. 成功基準 (第1スパン完了の定義)

1. `/housing` で **上部タブ6つ**が出て、URLで各ページに遷移できる（中身は探す=完成、他=既存 or プレースホルダ）。
2. **探すページ**が参考UIの洗練度で、実Firestoreデータ・フィルター・♡・ツアー追加・トレイ操作が**実際に動く**。「開始」でツアーが組成され `/housing/tour` へ遷移する（ナビ画面本体は次スパン）。
3. 質感=動画+ガラス+ハニー、ハードコードなし（`rgb(`/`rgba(`/`#hex`/`px;` grep がコンポーネントに残らない）。
4. `npm run build`（tsc厳密）+ `vitest run` が緑。[[feedback_vercel_tsc_strict]]
5. 体感で「サクサク」（グリッド描画で明らかなカクつきがない）。

## 10. 保留・次スパンへ送る論点
- 地図 B案の**同一エリア区違い同plot衝突**（オフセット or 許容）→ ツアー中スパン。
- **最寄りエーテライト**を型に持たせるか（登録UIには任意項目・GPT画像にも "開始エーテライト"）→ 登録/ツアー中スパン。
- 世界観の濃さ（動画+ガラスをどのページでどれだけ効かせるか＝「C:画面で使い分け」）→ 各ページ実装時に実画面で判断。
- 詳細ページをシェル内 Outlet にするか独立ルートか → 詳細スパン。
