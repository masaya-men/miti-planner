## 確定した設計方針（2026-03-28 第27セッション追記）

### 管理基盤・Firestore移行の設計方針
- 全ゲームデータ（コンテンツ・テンプレート・スキル・ステータス・アイコン・DC/サーバー）をFirestoreに段階的移行
- 操作中のFirestoreアクセスゼロ（起動時1回フェッチ→localStorage+メモリキャッシュ→同期的に使用）
- バージョン番号でキャッシュ無効化（変更時のみ再取得）
- マスターデータは種類ごとに1ドキュメント（読み取り回数の最適化）
- 管理画面は `/admin` ルート（同一アプリ内、Custom Claims認証）
- テンプレート自動生成: FFLogsインポートから発見フェーズ（14日）→ロック → 人気プラン昇格（管理者承認制）
- アイコンはFirebase Storage + Vercelエッジキャッシュで配信
- Firebase App Check + APIレート制限 + 監査ログ + BAN機能
- ローカル開発は本番Firestoreに読み取り専用接続（静的ファイルはフォールバック用）
- 詳細設計書: `docs/superpowers/specs/2026-03-28-admin-master-data-platform-design.md`

### プラン複製機能の設計方針
- サイドバーのプラン名横にコピーアイコン（既存の鉛筆マークの隣）
- ツールチップ: JA「すぐ下にコピーを作成」/ EN「Create a copy just below」
- ワンクリックで直下に「元の名前 (2)」として即座に複製（モーダルなし）
- プラン件数制限（1コンテンツ5件/合計50件）は通常の新規作成と同じチェック

### GoogleログインPWA対応の設計方針
- PWA（ホーム画面から起動）時のみ `signInWithRedirect` に切り替え
- PC版ブラウザは `signInWithPopup` のまま変更なし
- `window.matchMedia('(display-mode: standalone)')` で判定

## 確定した設計方針（2026-03-28 第26セッション追記）

### フリープラン（ダンジョン・レイド・その他）の設計方針
- 零式・絶はcontents.jsonのデータから選択、それ以外はユーザーが名前を自由入力
- 入力した名前がcontentIdとtitleの両方になる（1回の入力で済む）
- SavedPlanにcategoryフィールドを追加（サイドバーのグルーピングに使用）
- サイドバーではカテゴリ別に表示（零式/絶のようなシリーズ→階層の3段階ツリーではなく、カテゴリ→プラン名のフラット表示）
- Firestoreにもcategoryを保存・復元する

### NewPlanModalの設計方針
- レベル・カテゴリは常に未選択スタート（サイドバーからの直接作成は従来通り）
- コンテンツ選択はドロップダウンではなく1列フラットリスト（すぐ選べる）
- 未入力項目があれば案内メッセージ表示 + 作成ボタン無効
- Enterキーで作成実行可能

## 確定した設計方針（2026-03-28 第25セッション追記）

### サイドバー・ヘッダー線統一の方針
- glass-tier3クラス定義は触らない（23箇所で使用）
- 個別要素にglass-border-*-0で辺ごとにborderを除去する方式
- backdrop-filterのblur境界は要素を1pxオーバーラップさせて隠す
- ヘッダーハンドル上辺の微細なblur重なりはCSS構造的限界として許容
- 対象はサイドバー枠線・ヘッダー枠線・両者の接合部のみ。タイムラインは触らない

## 確定した設計方針（2026-03-27 第24セッション追記）

### ガラス表現の視認性向上方針
- タイムライン枠: ボーダー光沢 + 影（ダーク:グレー線+にじみ、ライト:黒パキッと線）。blurは不要
- ヘッダー区切り線: 今より明るくするだけ。影やglowは不要
- サイドバー選択中: 控えめなボーダー追加。枠として認識できる程度
- モーダル: 既にglass-tier3でそこそこ。ボーダーだけ合わせてもいい
- 全部に同じスタイルではなく、場所に合った強さにする（ごちゃごちゃ防止）
- テーブル横罫線: ユーザーがトグルで選択可能（デフォルト非表示）
- 光の周回アニメーションは軽減表ページではやらない（作業中ずっと見る画面なので集中の邪魔）

### サイドバーのハイライト階層ルール
- 開いているプラン: 左インジケーター（border-l-2）で「今ここ」を明示
- アクティブなプランを含む大カテゴリ: 全周ボーダーで強調
- ただ展開しているだけの大カテゴリ: グレーハイライトのみ
- ヘッダー区切り線: ダーク=白25%、ライト=純黒

### パフォーマンス最適化の順序
- 視覚的な変更を全部終えてから最後にやる
- 理由: UIの見た目を変えるとコンポーネント構造やpropsが変わる。先にメモ化すると調整し直す手間が増える
- React.memo / useMemo で対応予定

## 確定した設計方針（2026-03-24）

### サイドバーの操作フロー
- コンテンツクリック = 即編集開始。「テンプレート」という概念はユーザーに一切見せない
- テンプレートがあるコンテンツ → 裏で読み込み → 自動でプランとして保存 → 即編集
- テンプレートがないコンテンツ → 空プランを自動作成 → 即編集
- 同コンテンツに既存プランがある場合 → 即オープン（重複作成しない）
- 同コンテンツ1件以上 → サブアイテム展開。末尾に「+ 追加」行
- プラン名はサブアイテムのペンシルボタンでインライン編集（ヘッダーではダブルクリック）
- コンテンツ未紐付きプラン（自由作成）はレベル別「カスタム」セクションに表示
- 「新規作成」ボタンはレベルタブの上に控えめ配置

### 保存の仕組み（2層構造・確定方針 2026-03-30 第51セッション改定）
- **1層：localStorage（リアクティブ・無音・コストゼロ）**
  - 変更検知→500msデバウンス→即保存
  - 保存インジケーター（「保存中...」→「保存済み ✓」）はlocalStorage保存完了に連動
- **2層：Firestore（イベント駆動 + 定期バックアップ・ログインユーザーのみ）**
  - 保存タイミング：ページ離脱 / タブ非表示 / プラン切替 / ログアウト / 5分ごとの定期同期
  - 3分クールダウンで過剰書き込みを防止、forceSyncAllは10秒タイムアウト付き
  - 編集のたびにFirestoreに送るのはNG（試行錯誤中の無駄な同期を避ける）
  - ログイン時にカウンター修復（repairPlanCounts）+ ローカルが新しいプランはFirestoreに書き戻し
  - ログアウト時は全プラン強制同期（forceSyncAll）

### 共有の仕組み
- POST /api/share → Firestore → nanoid 8文字短縮URL → /share/:shareId
- PUT /api/share → 既存shareIdのロゴを上書き更新（第39セッション追加）

### デザイン方針
- まず全体を白黒ベースに統一 → あとでアクセントカラーを入れる
- ヘッダー: コンテンツ名は省略なし、プラン名を省略

### モバイル設計方針
- 全パネルはMobileBottomSheet方式、ボトムナビ常に最前面
- コントロールバーはモバイル非表示（ツールシートに集約）
- 軽減追加は5列フラットグリッド、連続配置可
- ダメージ表示はモバイルで千単位省略（211k / 1.5M）
- PC版には一切影響させない

### プラン管理方針
- 1コンテンツあたり最大5件 / 合計最大50件
- 30件超過でアーカイブ警告、50件でハードブロック

### FFLogs API管理方針
- ログイン限定、クライアント1時間15回制限
- APIキー5組ラウンドロビン運用中

### ユーザーフロー
- プラン0件: サイドバー開、オーバーレイ案内、コンテンツ選択でチュートリアル発火
- プランあり: サイドバー開閉は前回の状態を記憶
- 共有リンク訪問者: チュートリアルは出ない、自分で作ろうとした時に発火
