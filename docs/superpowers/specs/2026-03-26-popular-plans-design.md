# 軽減表人気ページ 設計書

## 概要
管理不要で、野良でよく使われている軽減表が自然とみんなに広まる仕組み。
共有プランのコピー数を追跡し、人気順で表示する独立ページ。

## ページ基本情報
- **URL**: `/popular`
- **別タブで開く**: コントロールバー・ツールシートからは `window.open('/popular', '_blank')`
- **多言語対応**: 既存i18n（`useTranslation()`）を使用
- **テーマ**: 既存CSS変数でダーク/ライト両対応

## アクセス導線
- **PC**: コントロールバーの「自分のジョブをハイライト」ボタンの左隣にボタン配置。デザインは既存ボタンと統一
- **スマホ**: ツールシート内にリンク配置

## 対象コンテンツ
- **最新零式**: M1S〜M4S（各コンテンツにつき1位・2位）
- **全絶コンテンツ**: 各コンテンツにつき1位・2位

## 人気の判定方法
- **主指標: コピー数（copyCount）**
  - 共有プランがコピーされるたびに Firestore の `copyCount` を +1
  - 同一ユーザーの重複コピーは除外（localStorageに「コピー済みshareId」を記録）
- コピー数が最も信頼できる理由:
  - 閲覧数はリロードで水増しされる
  - コピー = 「実際に使おうとした」という強い意思表示

## ページレイアウト

### 零式セクション
- M1S〜M4S それぞれ1位・2位をカードで表示
- カード内容: コンテンツ名 + コピー数 + ジョブ構成アイコン + 個別コピーボタン
- **「1位をまとめてコピー」ボタン**: M1S 1位 + M2S 1位 + M3S 1位 + M4S 1位 の4プラン一括コピー
- **「2位をまとめてコピー」ボタン**: M1S 2位 + M2S 2位 + M3S 2位 + M4S 2位 の4プラン一括コピー

### 絶セクション
- 各絶コンテンツにつき1位・2位をカードで表示
- カード内容: コンテンツ名 + コピー数 + ジョブ構成アイコン + 個別コピーボタン

## 技術仕様

### API: `/api/popular`（GET）
- コンテンツIDのリストを受け取り、各コンテンツの `copyCount` 降順トップ2を返す
- Firestoreクエリ: `shared_plans` コレクションで `contentId` フィルタ + `copyCount` 降順 + limit(2)
- **キャッシュ**: `Cache-Control: public, s-maxage=900, max-age=300`（サーバー15分、ブラウザ5分）
  - 人気ランキングは頻繁に変わらないので十分

### コピー数の追跡（既存処理に追加）
- `SharePage.tsx` の `handleCopyToMine()` 内で:
  1. localStorageの「コピー済みリスト」を確認
  2. 未コピーなら Firestore の `shared_plans/{shareId}` の `copyCount` を +1
  3. localStorageに shareId を記録（重複防止）
- 書き込みコスト: コピー1回につきFirestore書き込み1回

### 対象コンテンツの判定
- 零式: コンテンツマスタデータから最新の零式（Savage）レベルを取得
- 絶: コンテンツマスタデータから全Ultimate を取得

### Firebaseコスト見積もり
- コピー時: 書き込み+1回/コピー（既存処理に追加するだけ）
- ページ表示: 読み込み約10〜15件（対象コンテンツ数 × 2）
- キャッシュにより実際のFirestore読み込みは大幅に削減
- **無料枠への影響はほぼゼロ**

## ボタンデザイン
- コントロールバーの既存ボタン（ハイライト等）と同じスタイル
- アイコン + ツールチップで表現（既存パターンに合わせる）
