# 表の情報列固定（PC・横スクロール時）設計書

- **日付**: 2026-06-18
- **対象**: PC（768px 以上）のタイムライン表
- **要望元**: ユーザー要望（2026-06-18・TODO.md「未着手・将来計画」§ Excel ウィンドウ枠固定相当）
- **状態**: 設計確定（動作仕様はユーザー承認済み）

## 1. 目的・背景

タイムライン表は、左に**情報列ブロック**（フェーズ／ラベル／時間／敵の攻撃／元のダメージ／軽減後ダメージ）、その右に**スキル列**（パーティメンバーごとのジョブ列＝軽減アイコンと色付きバーを置く場所）が並ぶ。

フルパーティ（最大8人）など列が多いと表が横に広がり、横スクロールが発生する。現状は**シート全体が横スクロール**するため、右のスキルを見ようとすると左の情報列（時間・攻撃名・ダメージ）も一緒に画面外へ消え、「今見ている軽減がどの攻撃の行か」が分からなくなる。

本機能は、横スクロール時に**左の情報列ブロックを画面に固定**し、**右のスキル列だけが情報列の下を通り抜けてスクロール**するようにする（Excel のウィンドウ枠固定相当）。

## 2. スコープ

### やること
- PC で、横スクロールしても情報列ブロック（6 列まとめて）が据え置きで残る。
- スキル列（ジョブ列のセル＋軽減アイコン＋色付きバー）だけが横スクロールする。
- 情報列は**常時固定**（トグルなし）。横スクロールが発生しない幅では見た目は不変＝無影響。
- スクロール位置が 0 より右のときだけ、情報列の右端に**控えめな影**を出す（境界の奥行き手がかり）。スクロールを左端へ戻すと影は消える。
- 色・影はデザイントークン（`--color-*`）経由。ライト／ダーク両対応。

### やらないこと（スコープ外）
- **モバイル（768px 未満）は対象外**。モバイルは別コンポーネント（`MobileTimelineRow`）で、スキルが各行内に 1 列インライン表示される別レイアウト。横並びジョブ列の横スクロールという構造を持たないため、本機能は不要。
- トグル UI の追加（常時固定で十分なため）。
- 固定範囲を一部だけにする案（例：時間＋攻撃名のみ）。情報列ブロック全部を固定する。
- 縦スクロール挙動・表展開ロジックそのものの変更（既存維持。固定との両立だけ確認する）。

## 3. 現状アーキテクチャ（実コード根拠）

横方向に同期して動く要素は **3 レイヤー**あり、いずれも「情報列領域（左）＋スキル列領域（右）」で構成される。

1. **コントロールバー** … `controlBarRef` / `#timeline-controls-inner`（[Timeline.tsx:2349-2356](../../../src/components/Timeline.tsx#L2349)）。左＝ツールバー（表を展開する／AA 追加／メモ／undo・redo 等）、右＝ジョブアイコン列（JobPickerRow）。
2. **ヘッダー** … `headerRef` / `#timeline-header-inner`（[Timeline.tsx:2577-2697](../../../src/components/Timeline.tsx#L2577)）。左＝情報列見出し（フェーズ／ラベル／時間／敵の攻撃／元ダメージ／軽減後ダメージ）、右＝`RecastRow`（リキャスト表示のスキル列領域）。
3. **本文（行）** … `scrollContainerRef` 内の `sheetContainerRef`。各行は `TimelineRow`（[TimelineRow.tsx:304-707](../../../src/components/TimelineRow.tsx#L304)）。

### 横スクロールの仕組み
- 本文は `scrollContainerRef`（`overflow-x: auto`）の**ネイティブ横スクロール**。
- ヘッダーとコントロールバーはスクロールコンテナの**外**にあり、`handleScrollSync`（[Timeline.tsx:1290-1315](../../../src/components/Timeline.tsx#L1290)）が `scrollLeft` を読み、`#timeline-header-inner` と `#timeline-controls-inner` に `transform: translateX(-scrollLeft)` を当てて本文に追従させている。
- 列幅は CSS 変数（`--col-phase-w` / `--col-label-w` / `--col-time-w` / `--col-mechanic-w` / `--col-counter-w` ×2）で定義（[index.css:1343-1377](../../../src/index.css#L1343)）。情報列の合計右端 = `--col-member-start`（[index.css:1370](../../../src/index.css#L1370)）。

### 行の構造
- 各行 `TimelineRow` は `position: absolute`（top で縦積み）＋ `flex`。子は順に：フェーズ／ラベル／時間／敵の攻撃／U.Dmg／Dmg（＝情報列）、続いて各メンバーのスキル列セル（中身は空のクリック用 `div`）。
- 行には性能最適化 `content-visibility: auto` ＋ `contain-intrinsic-size: auto 50px`（[TimelineRow.tsx:309-310](../../../src/components/TimelineRow.tsx#L309)）。

### 軽減アイコン・色付きバーのレイヤー
- スキルアイコン／色付きバーは**行とは別の絶対配置レイヤー**として `sheetContainerRef` 内に重ねて描画される（`MitigationItem`・[Timeline.tsx:3066以降](../../../src/components/Timeline.tsx#L3066)）。
- `MitigationItem` の外枠は `position: absolute; left: <絶対px>`。内部 z-index：色付きバー（エフェクト棒）= `z-10`（[Timeline.tsx:536-549](../../../src/components/Timeline.tsx#L536)）、アイコン = `z-20`（[Timeline.tsx:480-489](../../../src/components/Timeline.tsx#L480)）、ターゲットバッジ = `z-30`（[Timeline.tsx:522](../../../src/components/Timeline.tsx#L522)）。
- このレイヤーは JSX 上で**行より後**に描画される＝デフォルトの描画順で行より上に乗る。

## 4. 中核となる技術課題（重要）

「情報列セルに不透明背景＋高い z-index を当てるだけ」では**不十分**。理由：

- 行は `content-visibility: auto` を持つ。これは画面内のとき `contain: layout style paint` 相当の封じ込めを行い、**各行が独立した stacking context（重なり領域）になる**。
- 軽減アイコン／色付きバーのレイヤーは行とは別の兄弟要素で、DOM 上で**行より後**に描画される＝行（とその中の情報列セル）より上に乗る。
- 結果、情報列セルを行の中で z-index 最大にしても、**行ごと**別レイヤーの下に潜るため、横スクロールで左へ来たバー／アイコンが情報列の上にはみ出して見える。

→ 情報列を「**スキル列レイヤーより確実に上に描画される独立した固定レイヤー**」として扱う必要がある。これが本機能を「中規模」たらしめる核心。

### 採用方針：情報列を専用の固定レイヤーに分離（要 spike 確認）
- 情報列（6 列）を、スキル列の描画より**後**に置かれる sticky な独立レイヤーとして描画し、`position: sticky; left: 0`（または相当の固定）・**不透明背景**・スキルレイヤーより高い z-index を持たせる。これにより、スクロールしてくるバー／アイコンは情報列レイヤーの下に隠れる。
- ヘッダー・コントロールバーは既にスクロールコンテナ外で `translateX` 管理されているため、**情報列領域だけ `translateX` 対象から外し（据え置き）、スキル列領域（RecastRow / JobPickerRow）だけを `translateX` する**ように同期処理を分割する。

### 実装着手前の spike（短時間の検証）
実装の最初に、以下を実機で確認して最終方式を確定する（`content-visibility` と stacking の相互作用が方式選択を左右するため）：
- (案A1) 情報列セルを `position: sticky` 化し、行の stacking を軽減レイヤーより上げる方式が、`content-visibility` 下でバーのはみ出し無し・かつアイコンのドラッグ等の pointer-events を壊さずに成立するか。
- (案A2) 情報列を本文内の**独立 sticky レイヤー**として分離描画する方式（A1 が成立しない場合の本命）。
- spike の結論を本設計書に追記してから実装に入る。

## 6. 影（境界）の見た目
- スクロール位置 0：影なし。
- スクロール位置 > 0：情報列ブロックの右端に控えめな影（右方向へ短く減衰）。
- トークン経由・ライト／ダーク両対応。`box-shadow` で実装し `clip-path: path()` は使わない（CSS ルール準拠）。

## 7. QA チェックリスト（実機・ユーザー確認）
1. 右端まで横スクロール → 情報列ブロックが据え置きで残る／スキル列だけ動く。
2. スクロール > 0 で右端に影が出る／左端へ戻すと影が消える。
3. **色付きバー（青・オレンジ）・軽減アイコンが情報列の上に透けて出ていないか**（上端・下端まで、複数行にまたぐ長いバーで確認）。← 最重要。
4. 「表を展開する」（`hideEmptyRows`）切替時に固定がズレない／縦横スクロール併用で破綻しない。
5. フェーズ区切り・ホバー罫線・TL 選択ハイライトが情報列の上で正しく描画される。
6. ライトモードで影・不透明背景・境界線が破綻しない。
7. DPR 2.58（開発者画面・CSS 幅約 1489）と一般的な 1920 幅の両方で列がズレない。
8. モバイル（768px 未満）で見た目・挙動が一切変わっていない（無影響）。
9. 軽減アイコンのドラッグ移動・セルクリックでの追加が従来どおり動く（pointer-events 退行なし）。
