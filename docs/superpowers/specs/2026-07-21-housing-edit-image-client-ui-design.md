## 前提認識の確認 (brainstormingで訂正した点)

「複数投稿URL機能(Batch 2)」と「画像とURLを同時に持たせる」は、"まずA→次にB" という決まった順番の計画ではない。両方とも独立した将来検討事項で、優先順もタイミングも未定(Plan A設計書の「含まない」節参照)。今回のPlan Bはどちらとも直接の順序関係を持たない。

## 基本方針: 専用の「差し替え」UIは作らない

Plan Aで作ったサーバーAPIは「1枚削除する」「新しい1枚を追加する」「並び替える」の3つのみで、「差し替え」専用のエンドポイントは存在しない。したがってクライアント側でも「差し替えボタン」は作らず、**既存の✕(削除)ボタン + 上限未満のときに出る「追加」UI** の組み合わせで代替する。画質の悪い写真を良い写真に入れ替えたい場合は「✕で消す→追加で足す」の2手順で行い、元の並び順(特にカバー画像)を保ちたい場合はドラッグで並び替える。

これにより、新規登録フォームで既に実装済みの `HousingRegisterImageField` / `HousingRegisterSourceImageUrlsField` の削除・追加UIをほぼそのまま編集モードでも使い回せる (新しい見た目の作り込みが最小限で済む)。

## 確認ダイアログは一切なし

以下の操作はすべて確認ダイアログを挟まず、ボタン一発で実行する:

- 直接アップロード画像の削除
- URL経由画像の削除
- 登録方法 (アップロード⇔URL) の切り替え

理由 (ユーザー判断): 新規登録フォームの✕ボタンも元々確認なしであり挙動を統一したい / 削除してもユーザーのPCから消えるわけではなく編集画面を開いている本人がもう一度アップロードし直せば復旧できる / 他サービスでも「えー最初からかよ」となる過剰な確認は体験として良くない、との判断。

## 保存中の見た目: 「少し待ってから反映」方式 (B案)

即時サーバー反映のため、ボタン操作からサーバー応答までに一瞬のラグが生じる。UIは「押した瞬間に消える」のではなく、**処理中であることが分かる見た目(グレーアウト等、既存のドラッグ中 `opacity:0.4` と同系統の表現を流用)にしてから、サーバー応答が返った時点で確定反映する**。失敗時は処理中表示を解除して元の状態に戻す。

- **削除**: ✕を押すと対象タイルが処理中表示になり、サーバー応答後に一覧から消える。
- **追加 (アップロード)**: ファイル選択→既存の圧縮中表示(そのまま流用)→圧縮完了後にアップロードAPI呼び出し→処理中表示→成功したら通常タイルとして確定。
- **並び替え**: ドラッグ中の見た目 (dnd-kitの標準挙動) は変更しない (ドラッグ中に並びが動くのは既存動作そのまま)。ドロップした瞬間に並び順を確定表示しつつ、裏で並び替えAPIを呼ぶ。応答が来るまでは軽い処理中インジケータを出し、失敗時は元の並び順にロールバック + エラー通知。

並び替えAPIはドラッグ完了 (ドロップ) の瞬間に1回だけ呼ばれる (ドラッグ中に連続で呼ばれるわけではない)。ユーザーから「コストが跳ねないか」の懸念があったが、通常のFirestore書き込み1回分のコストであり、通常の利用範囲では問題にならないことを確認済み。

## エラー時の表示

既存の `showToast` (画面上部の通知、成功時に緑で使われている仕組み) をそのまま流用し、失敗時は同じ場所に「失敗しました。もう一度お試しください」を出す。原因ごとに個別のエラーメッセージを出し分ける作り込みは今回はしない (ユーザー判断: 実際に起きる失敗のほとんどは通信の一時的な不調で再試行すれば直ることが多く、それ以外の特殊ケースはまず起きないため過剰な作り込みは避ける。実運用で「再試行しても直らない」との報告が出たら個別対応を追加する)。

## 登録方法の切り替え (アップロード⇔URL): タブはローカル表示切り替えのみ

**重要な発見**: サーバー側の検証 (`validateImage`、`src/utils/housingValidation.ts:324`) は `imageMode==='sns'` を送る際に「Tweet/YouTube/sourceImageUrls のいずれか1つ以上」を必須としている (`source_required_for_sns` エラー)。そのため、タブを押した瞬間に「imageMode切り替え」を伴う更新をサーバーへ即送信する設計では、まだ何も入力していない段階でエラーになってしまう。

**対応方針**: タブ切り替えは**ローカルの表示切り替えのみ**とし、実際にサーバーへ反映する (= 旧方式のデータをクリーンアップする) のは、**新しいモードで実際に有効なコンテンツが1つ入った瞬間**にする。

- 「URL」タブに切り替えただけの時点では何もサーバーに送らない。実際にURLを貼って画像/動画の取得に成功した瞬間、そこで初めて `update-listing` を呼び `imageMode:'sns'` + 取得したデータを保存する (このタイミングで旧アップロード画像のクリーンアップも発生)。
- 「アップロード」タブに切り替えただけの時点では何もサーバーに送らない。実際に1枚アップロードが完了した瞬間、そこで初めて `imageMode:'thumbnail'` への切り替え + 旧SNSフィールドのクリーンアップを伴う更新を行う。

この方式なら、タブを押し間違えた・押してから気が変わった、というだけでは何も消えない。確認ダイアログなしの「サクサク」操作感を保ちながら、誤操作でデータが消える事故を防げる。

## 画像0枚の状態: 特別な作り込み不要

物件は元々「画像もURLも一切なし (`imageMode:'none'`)」で登録できる (画像は必須項目ではない、既存仕様)。上記のタブ設計により編集中に意図せず0枚になることは無いが、**そもそも0枚で登録された既存物件を編集で開くケースは元からあり得る**。

この場合、既存の `HousingRegisterImageField` (0枚なら「追加」の枠のみ表示) / `HousingRegisterSourceImageUrlsField` (0枚なら何も描画せず、SNS URL入力欄だけが常に表示される) がどちらも0件表示を自然にハンドリングできる作りになっているため、新規登録フォームと全く同じ体験がそのまま成立する。追加の実装は不要。

## RegisterPage.tsx 側の見直し箇所 (「方式A」撤廃に伴う)

現状、編集モードは「写真セクション自体を表示しない」方式A ([RegisterPage.tsx:201](../../../src/components/housing/pages/RegisterPage.tsx)) を前提に以下が実装されている。Plan Bではこれらを全て見直す:

- `visibleStepIds(mode)`: `mode==='edit'` のとき `media` ステップを除外している → 除外をやめ、ステッパーに「写真」ステップを復活させる。
- `checklistItems` の `imageRequired: mode==='create'`: 編集でも画像行を評価対象にする (ただし「必須」にするかは要検討。現状は新規登録のみ画像必須、編集は任意のままで問題ない = 既存の0枚許容の物件と矛盾しないため `imageRequired` は `mode==='create'` のまま据え置きでよい)。
- `checkPanelItems` で `mode==='edit'` のとき image 行を filter で除外している → 除外をやめる。
- `doneMap.media` で `mode==='edit'` のとき常に `true` 扱いにしている → `hasMedia` の実値を使うようにする。

## 案内文 (URL経由登録を勧める文言) は実装時に確定

Plan A設計書のスコープ5番で触れている「URL経由登録を勧める案内文」の正確なコピーは、実装フェーズ (writing-plans/実装エージェント) で下書き→本セッションのようにユーザー確認を経て確定する。本書では「画質が劣化しない」「Twitterの場合は元投稿削除で連動して消える」という2点を伝える方向性のみ確定させる。

## テスト方針 (Plan A設計書を継承・クライアント側を追記)

- クライアント (vitest): タブ切り替え (ローカルのみ・サーバー未送信であること) / 削除ボタンの処理中→確定の状態遷移 / 並び替えの確定→失敗時ロールバック / 「最後の1枚」で削除ボタンがdisabledになること / 0枚状態からの新規追加。
- 実機確認 (ユーザー依頼): 直接アップロード⇔URL切り替えの往復・削除で詰まることの目視・並び替えの目視・処理中表示の見た目。

## リスク・影響範囲チェック

- 既存の「方式A」を前提にしたコード・テストの見直しが必要 (上記「RegisterPage.tsx 側の見直し箇所」参照)。範囲は特定済みで想定外の波及は無い見込み。
- 差し替え専用UIを作らないことで、Plan A側に新規エンドポイントの追加要求は発生しない (既存の delete/upload API のみで完結)。
- タブ切り替えを「実際にコンテンツが入るまでサーバー未反映」にすることで、Plan Aで実装済みのクリーンアップロジック (Task7) の呼び出しタイミングが「タブ押下時」ではなく「URL取得成功時 / アップロード完了時」に変わる。Plan A側のAPI仕様・データ整合性には影響しない (呼ぶタイミングが変わるだけで、呼ぶ内容・保証は既存のまま)。
