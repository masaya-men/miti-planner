# プロジェクト概要: FF14 Housing Tour Planner（仮）
従来の「文字ベースの住所リスト」から脱却し、直感的なマップUIと美麗な画像ギャラリーで「ハウジングを巡る体験（ツアー）」そのものをデザインする、FF14専用の次世代ハウジングポータルWebアプリ。

## 1. アーキテクチャと完全無料のデータ設計
- **フロントエンド:** [React / Vue / Svelte などから指定] + Tailwind CSS
- **データベース:** Firebase Firestore（Authによるユーザー認証、リアルタイムDB含む）
- **画像の取り扱い（コスト・パフォーマンス最適化）:**
  - サーバー代を無料に保つため、画像データ自体は自社データベースに一切保存しない。
  - ユーザーが入力したTwitter（X）のURLから、フロントエンド側で「OGP画像のURL文字列」のみを抽出し、Firestoreには軽量なテキストデータ（DC, サーバー, 住所, 画像URL, タグ等）のみを保存する。
- **大規模アクセス（バズ）対策と負荷分散:**
  - **課題:** W1stチームや有名配信者が「閲覧専用URL」をSNS等で公開し、数万人が同時に1つのデータにアクセスした場合、Firestore（データベース）の同時読み込み制限や従量課金コストのスパイクが発生するリスクがある。
  - **解決策（エッジキャッシュの導入）:** スプレッドシートのようにクラッシュさせないため、閲覧専用URLからのアクセスに対してはデータベースへ直接アクセスさせず、CDN（VercelやCloudflare等）のエッジサーバーに一時保存された「キャッシュ（静的データ）」を配信するアーキテクチャを採用する。
  - **効果:** これにより、どれだけ突発的な膨大なアクセスが来てもサーバー負荷とコストを完全に抑え込み、ユーザーに「重い」と感じさせない快適な閲覧体験を維持する。


## 2. UI/UXデザインとグローバルテーマ連動
- **グローバルテーマの同期:** 総合ポータルとして、Miti Planner（軽減表）とテーマ設定（ライト / ダーク）を共有・同期し、アプリ間移動時の視覚的ストレス（急激な明暗差）を完全に排除する。
- **背景画像のデフォルト固定（設定の排除）:**
  - ユーザーによる背景画像のカスタマイズ機能はあえて実装せず、アプリ側で用意した「世界観に合い、かつ文字の視認性を下げない（あらかじめ少しボケている）抽象的なデフォルト風景画像」に固定する。
- **テーマ別のグラスモーフィズム設計（モダン・アブストラクト背景）:**
  - アプリのブランドイメージを統一し没入感を最大化するため、背景はゲーム内の風景写真ではなく、Miti Plannerと共通の「モダンで抽象的なテクスチャ」を採用する。
  - **ライトモード（昼景）:** 完全な純白（#FFFFFF）に微かな水滴とプリズム（淡い青や緑）の光が滲む抽象背景に、半透明の白パネル（すりガラス）を適用。
  - **ダークモード（夜景）:** 完全な漆黒（#000000）に微かな水滴と一筋の鋭い光が反射する抽象背景に、半透明の黒パネル（ブラック・グラス）を適用。

- **画面遷移ゼロのUX:** ギャラリーからのお気に入り追加やツアーへの組み込みは、ページ遷移を一切挟まず、シームレスなアニメーションで左側のサイドバーに格納される仕様とする。
- **モバイル・ファーストなレスポンシブ設計（スマホUI最適化）:**
  - **ボトムナビゲーション:** スマホ版では画面下部に「🏠 探す(ギャラリー)」「🗺️ 回る(ツアー)」「➕ 登録」等のメニューを固定配置し、アプリライクな片手操作を実現する。
  - **ツアー画面のボトムシート構造:** ツアー実行中は、背景を全画面のマップ表示とし、画面下部から引き出せる「ボトムシート（半透明の黒パネル）」にツアー先のリストを格納する。スワイプでリストを引き上げ/下げできる直感的な操作感とする。
  - **直感的な「次へ」アクション:** ボトムシート上部にフローティングの「↓ 次のハウジングへ」ボタンを配置し、親指のタップだけでサクサク次の目的地へマップを切り替えられる設計にする。




## 3. ギャラリー表示とマップ連携機能
- **爆速ギャラリー:** OGP画像URLを `<img>` タグで表示し、遅延読み込み（Lazy Load）を実装。数百枚並べても60fpsの軽快なスクロールを維持する。
- **直感アクション:** 写真にホバー（タップ）すると、「❤️（お気に入り）」と「📍 今日行く（ツアー追加）」ボタンが浮かび上がる。
- **インタラクティブ・マップ:** - ツアー開始時、右画面にFF14のハウジングマップを再現した地図を表示。
  - 選択した複数の家の番地データに基づき、「最も近い簡易エーテライト」を算出し、最短の徒歩ルートをマップ上に視覚的に描画する。

## 4. アカウント同期とゲストへの配慮（クロスデバイス体験）
- **ゲストユーザー（未ログイン状態）:**
  - 「📍 今日行く」で作成したツアールートは、ブラウザのキャッシュ（LocalStorage）に保存し、ログイン不要でツアー機能を利用可能にする。
  - サイドバー下部に「💡 スマホで作ったルートをPCに引き継ぎたい場合は、ログイン（無料）してください」と自然な誘導を配置する。
- **ログインユーザー（Firebase Auth）:**
  - お気に入りやツアールートをクラウド（Firestore）に同期。スマホのスキマ時間にルートを組み、帰宅後にPCの大画面でツアーを開始できるシームレスな体験を提供する。

## 5. 登録システムとクリエイター保護（離脱率0%設計）
- **スマート入力と手動補完:**
  - TwitterのURLをペーストすると、事前定義した `masterData.js` の正規表現を用いてDC・サーバー・エリア等を自動抽出・入力する。
  - 抽出漏れがあった場合は、プレビュー画面のドロップダウンから手動で修正可能。
  - URLがない場合は、地図上をクリックするだけで番地が入力されるマップ入力モードを用意する。
- **登録枠の管理（スパム対策と古参配慮）:**
  - **初期ボーナス:** アカウント作成時、古参クリエイターの離脱を防ぐため「20件」の初期登録枠を付与する。
  - **デイリー回復:** 以降は「1日3件まで」の登録制限を設け、毎日0時に3枠まで回復する（UI上で「今日の残り登録枠：3/3」と明示する）。
- **枠の返還（心理的負担ゼロ）:** 登録情報の編集は常時無料。「今日登録したデータ」をその日のうちに削除した場合、消費した枠が「1」回復するシステムとし、登録ミスの恐怖を無くす。
- **タグ機能:** 「モダン」「和風」等のテイストや季節タグを事前定義し選択させる。用途やマナーに関するタグ（RP、見学自由、スタジオ等）はトラブル防止のため実装しない。

## 6. ツアー進行機能とリアルタイム共有（キラー機能）
- **DC（リージョン）の自動分割:** ツアールートに追加された家は、左側のサイドバー内で自動的に「JP」「NA」などの物理DCごとのタブに分割され、別ツアーとして扱われる。
- **自動整列と始点指定:** - ルート内は「サーバー」ブロックごとにまとまり、「区 → 番地」の若い順に自動整列する。
  - 任意の家の「🏁ここからスタート」ボタンを押すと始点に設定され、以後の順番はドラッグ＆ドロップで自由に入れ替え可能。
- **洗練された進行UI:** マップ画面にフローティングで大きな「↓ 次のハウジングへ」ボタンを配置。PCユーザーは「Enterキー（またはSpaceキー）」を押すだけで次の目的地へ即座に切り替わる。
- **ツアーURLの共有と2つのモード:**
  - 発行された専用URLをSNS等で共有することで、他者を自分のツアーに招待できる。
  - **モードA（フリー探索）:** 参加者が各自の画面で自由に「次へ」を押して進めるモード。
  - **モードB（ツアーガイド連動）:** 主催者が「次へ」を押した瞬間に、URLを開いている参加者全員の画面がFirebaseのリアルタイム同期によって自動で次のマップへ切り替わるオンラインイベントモード。

## 7. エコシステムと自浄作用
- **マナー順守の明記:** ツアー開始ボタンを押した直後、マップ表示と共に「訪問時は家主の迷惑にならないようマナーを守り、SNS等のルールを優先してください」というポップアップを必ず表示する。
- **いいね（❤️）の完全非表示:** 人気格差を防ぐため、「❤️」はお気に入り（ブックマーク）機能としてのみ機能させ、他者からの獲得数は一切非表示にする。
- **コンテンツルーレット：ハウジングツアー:**
  - ユーザーが任意のタグを選択してボタンを押すと、全データからアルゴリズムがランダムに5件の家をピックアップし、一期一会のツアールートを自動生成する。
- **自浄作用（通報システム）:**
  - 「行ったけど家がなかった / 違う家だった」という通報が3件集まったデータは、自動でマップから非表示にする。
  - オーナーには非表示の旨を通知し、マイページから1クリックで復活可能にする。
  - 虚偽の通報による荒らしを防ぐため、裏側に「信用スコア」を実装。オーナーが復活させた場合、嘘の通報を行ったユーザーのスコアを下げ、以後の通報を無効化する（このロジックは非公開とする）。
