# ハウジンガープロフィール + ハウジンガーページ 設計書

> 2026-07-10 brainstorming でユーザー承認済み。公開前必須の大物3つの1つ目
> (原要望 = `docs/.private/2026-07-08-housing-release-feature-braindump.md` #3)。
> 実装は本設計書 → writing-plans の計画書を経てから。

## 1. 目的

- 登録者(ハウジンガー)が**任意で**名乗れるようにし、「自分の場所ができる」動機づけで登録を増やす。
- 見る側は「この人の他の家」をまとめて見られる。個人タグと合わせて「@名前 の家だけのツアー」の入口になる。

## 2. 確定済みの設計判断 (2026-07-10 ユーザー確定・変更不可)

1. **ハウジンガー名 = 個人タグ (一体化)**。名前を設定するとそれがそのまま個人タグになる。設定は1回・名前は1つ。
2. プロフィール項目 = **名前 + アイコン + ひとこと自己紹介 + SNSリンク**(名前以外は任意)。
3. 登録者表示は**詳細パネルのみ。探すカードには出さない**。
   理由 =「人気な人だけしか見てもらえないのは嫌」(一覧は家が主役・人気偏重を避ける)。
   この思想は今後の並び順・表示の判断にも適用する。
4. アーキテクチャ = **公開プロフィール中央方式**(案1)。listing への焼き込み(デノーマライズ)はしない。
5. 個人情報を持たない方針 ([[feedback_auth_privacy]]) は維持。ハンドルネーム + アイコンのみ。
   識別子はハッシュ化済み uid (`hashed:…`)。生 Discord ID はどこにも出さない。

## 3. データ設計

### 3.1 公開プロフィール: `housing_profiles/{uid}` (新設コレクション)

| フィールド | 型 | 説明 |
|---|---|---|
| `displayName` | string | 必須。公開時点の `users/{uid}.displayName` のコピー。個人タグのラベル源泉 |
| `avatarUrl` | string \| null | `users/{uid}.avatarUrl` のコピー(既存 Storage `users/{uid}/avatar.webp` の URL) |
| `bio` | string \| null | ひとこと自己紹介。最大文字数は定数化 (目安 100 文字) |
| `snsUrl` | string \| null | 許可ドメインのみ (§6.1) |
| `isPublished` | boolean | 公開中フラグ。false = 非公開 (§3.4) |
| `isModerationHidden` | boolean | 運営による強制非公開 (§6.2)。true なら isPublished に関係なく非表示 |
| `createdAt` / `updatedAt` | timestamp | サーバー時刻 |

- **読み取り**: 全員可 (firestore.rules に public read を追加)。
- **書き込み**: クライアント直書き禁止。**必ず API (サーバー) 経由**で idToken 検証 → 本人の uid のドキュメントのみ更新。既存 housing API ハンドラの認証パターン (`verifyIdToken` + App Check、`buildHousingHeaders`) を踏襲。

### 3.2 アカウント設定との連動 (設定箇所は1つ)

- 名前・アイコンの編集 UI は既存のアカウントモーダル (`HousingAccountModal` / `useAccountActions` / `AvatarCropModal`) を流用。**新設しない**。
- 公開 ON の間の追従コピー: 現状の名前・アイコン更新は**クライアントが `users/{uid}` を直接更新**する作り
  (`useAuthStore.updateDisplayName` / `avatarUpload.ts`)。ここに「更新成功後、公開中なら**プロフィール同期 API を呼ぶ**」を追加する。
  同期 API はサーバー側で `users/{uid}` の現在値を読んで `housing_profiles/{uid}` に転記する (クライアントから値は受け取らない = 改ざん不可)。
- 一括データ改変は行わない。更新対象は常に本人の 1 ドキュメントのみ ([[feedback_silent_data_mutation]] 抵触なし)。

### 3.3 個人タグとの一体化

- **個人タグの実体 = uid 参照**。listing に保存するのはタグ ID (**uid から決定的に導出**。名前由来スラッグ禁止 = 改名で ID が変わらない)。
- 名前の**源泉は `housing_profiles/{uid}.displayName` の 1 箇所**。タグ刷新 Phase B の `personal_tags/{tagId}.displayName` は
  デノーマライズドコピーとして残してよいが、**更新はプロフィール同期 API が同一サーバー処理内で一括転記**する
  (クライアントや別経路からの個別更新を禁止)。→ 名前変更が即タグに反映され、ズレが構造的に起きない性質は維持。
- タグ刷新 Phase B (`docs/superpowers/plans/2026-07-10-housing-tag-overhaul-plan.md`) との整合:
  - TagPicker「個人」タブの「自分のタグ作成」= **ハウジンガー公開への導線**に置き換える (名前公開 = タグ誕生)。
  - Phase B が本設計より先に完成した場合、統合時に上記へ調整する (統合担当 = 親セッション)。
- プロフィール未公開のユーザーに個人タグは存在しない (作成 UI は公開を促す)。

### 3.4 非公開化の挙動

- 公開 OFF (または運営の強制非公開) で: 詳細パネルの登録者行が消える / ハウジンガーページは NotFound /
  個人タグが絞り込み候補・表示から消える (listing 内のタグ ID は残置し、解決失敗 = 非表示)。
- 登録済みハウジング自体には一切触れない。消えるのは「誰が登録したか」の表示だけ。

## 4. 画面設計

### 4.1 設定導線 (2箇所・中身は同一)

1. **アカウントモーダル** (本籍地): 「ハウジンガーとして公開」トグル + ひとこと + SNSリンク入力を追加。
2. **登録フォーム末尾** (任意ブロック): 未公開の人に「ハウジンガーとして名乗りますか?(任意)」を表示。
   その場で名前・アイコン設定 (既存部品流用)。公開中の人には「〇〇として公開中」の小さな表示のみ。
   - 登録フォームは `feat/housing-register-improvements` (ステッパー改修) の新構成に合わせて組み込む。

### 4.2 詳細パネル

- 登録者行を1行追加: 小アイコン + 名前。クリックでハウジンガーページへ。
- ひとこと・SNSリンクは詳細には出さない (詳細は家が主役)。
- プロフィール読み込み失敗・未公開時は行ごと非表示 (レイアウトを占有しない)。

### 4.3 ハウジンガーページ `/housing/housinger/:uid`

- ルートは `App.tsx` の housing 子ルートに追加 (`listing/:listingId` と同型)。URL はシェア可能。
- 構成 (上から): アイコン(大) + 名前 + ひとこと + SNSリンク / その人の**公開**ハウジング一覧 (探すのカードグリッド流用) /
  「この人の家をまとめてツアーに追加」ボタン / 控えめなメニューに通報。
- 本人閲覧時のみ「プロフィールを編集」ボタン (アカウントモーダルを開く)。本人でも見えるのは公開分のみ (v1)。
- 未公開・存在しない uid は housing トンマナの既存 NotFound を流用。
- 一覧クエリ: `where('ownerUid','==',uid)` + 公開条件。**firestore.indexes.json への複合インデックス登録を必ず確認**
  ([[reference_firestore_composite_index]] 漏れると本番で沈黙する)。

### 4.4 探すとの接続

- 個人タグで絞り込み中、結果一覧の上に「〇〇のハウジンガーページを見る」リンクを小さく表示。

## 5. トンマナ・i18n

- `.claude/rules/housing-design.md` 準拠 (装飾999pxピル/色付きalert箱/過剰glow 禁止)。正典 = housing-tour-mockup。
- 文言は ja/en/ko/zh 4言語 parity。ロケール JSON はブロック単位 textual 編集。
- 新規文言は「ハウジング」統一 (「物件」禁止)。名乗りの呼称は「ハウジンガー」。

## 6. 安全設計

### 6.1 SNSリンク検証

- 許可リスト方式 (定数 `HOUSINGER_SNS_ALLOWED_HOSTS`): v1 = **X (x.com / twitter.com) / YouTube (youtube.com / youtu.be) / Lodestone (jp.finalfantasyxiv.com 等公式ドメイン)**。
- https のみ・1人1個。検証は**サーバー側**で実施 (クライアント検証は補助)。リスト外は保存時に拒否。
- 表示時は `rel="noopener noreferrer"` + `target="_blank"`。

### 6.2 通報・モデレーション

- 通報対象種別に **profile** を追加 (現行は listing のみ)。理由 = 不適切な名前・アイコン / なりすまし / その他。
- 既存の通報 UI (`HousingReportModal`) と管理画面キューを流用し、対象種別で分岐。
- 運営アクション = **管理画面から強制非公開** (`isModerationHidden`)。/admin で完結 ([[feedback_housing_admin_complete]])。
- listing 通報にある閾値自動非表示は**プロフィールには適用しない** (嫌がらせ通報で無実の人が消える方が害が大きい。v1 は人の目で判断)。

### 6.3 公開条件・エッジケース

- 公開 ON の必須条件 = **名前のみ** (名前 = 個人タグのため)。アイコン未設定は既存の初期アバター表示。
- 同名ハウジンガーは許容 (内部は uid で区別)。絞り込みに同名タグが並ぶのは v1 許容。
- プロフィール取得はセッション内キャッシュ (同一 uid の重複読み取りを抑制。Firebase コスト配慮)。

## 7. テスト方針

既存の housing テストパターン (vitest) に従い、最低限:
1. SNS URL 検証 (許可/拒否/https 強制) — 純関数ユニット
2. 公開条件 (名前必須・非公開時の解決失敗挙動) — ユニット
3. 詳細パネルの登録者行 (公開時表示 / 未公開・失敗時に行ごと消える) — コンポーネント
4. ハウジンガーページ (公開一覧のみ表示・本人時の編集ボタン・NotFound) — コンポーネント
5. API ハンドラ (本人以外の書き込み拒否・追従コピー) — 既存ハンドラテストの型に従う

## 8. スコープ外 (YAGNI・明示的に作らない)

- フォロー / お気に入りハウジンガー / 人気順・実績数の表示 (§2-3 の思想により意図的に排除)
- マイページタブの本実装 (`mypage` は ComingSoon のまま)
- プロフィールの自動非表示閾値 / Reporter scoring (モデロードマップ側で将来検討)
- SNSリンクの許可ドメイン拡張 (定数追加のみで可能な構造にしておく)
- 探すカードへの登録者表示 (§2-3 で不採用と確定)

## 9. 実装順の目安 (計画書化の入力)

1. データ層: `housing_profiles` + rules + API (公開/更新/非公開 + 追従コピー) + SNS 検証
2. 詳細パネルの登録者行 + ハウジンガーページ (ルート/一覧/NotFound) + 複合インデックス
3. 設定導線 (アカウントモーダル拡張 → 登録フォーム任意ブロック)
4. 個人タグ接続 (タグ刷新 Phase B との統合調整) + 探すの接続リンク
5. 通報 profile 対応 + 管理画面の強制非公開
