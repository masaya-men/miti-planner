# リアルタイム共同編集 設計書 (2026-06-03)

> 1 枚の軽減表 (`PlanData`) を複数人がリアルタイムに同時編集できるようにする。
> 業界標準 (Yjs / CRDT) を、すでに使っている Cloudflare 上で、無料プランの $0 ハードストップ内に収まるよう実装する。
> ブレインストーミング (2026-06-03) の合意を spec 化したもの。前提メモ: [docs/.private/2026-06-03-realtime-collab-and-sync-notes.md](../../.private/2026-06-03-realtime-collab-and-sync-notes.md)

---

## 1. ゴールと非ゴール

### ゴール
- 複数人が **同じ 1 枚の軽減表**を、**同時に編集**し、お互いの変更が**ライブで見える** (Google Docs 体験)。
- 編集権の取り合いで一方の変更が消えない (CRDT による衝突なしマージ)。
- 誰が在室し、どこを指しているかが見える (presence / カーソル)。
- **コストが青天井にならない** — 最悪値が事前に計算でき、無料プランの $0 ハードストップを基本線にする。

### 非ゴール (YAGNI / 今回作らない)
- 版管理・編集履歴の巻き戻し (将来、保存層が既存処理を通るので後付け可能)。
- 参加の 1 人ずつ承認制 (リンクを持つ人は編集可、で確定)。
- オーナーによる強制キック / 行ロック (最初は「セッション終了」だけ)。
- 自分の複数端末間の同期改善 (現状で不便なし。**触らない**)。
- 編集ロック / 持ち回り方式 (CRDT で不要。**採用しない** — 後で全捨てになるため)。

## 2. 全体アーキテクチャ — 「会議室のホワイトボード」モデル

```
[本棚 = 既存の保存]                 [会議室 = ライブ部屋]
 Firestore のプラン   ──①seed──▶   Cloudflare Durable Object
 (PlanData 1 件)                     内の Yjs ドキュメント
       ▲                                 │   ▲   ▲
       │                          編集者A 編集者B 閲覧者C
       │                          (編集=WebSocket / カーソル=P2P)
       └──②書き戻し (既存 toFirestoreUpdate)──┘
         (部屋クローズ時 + 数分間隔の安全保存のみ)
```

- **部屋 = Durable Object (DO) 1 個**。中に 1 枚の `PlanData` を表す Yjs ドキュメントを保持。
- **①seed**: 最初の参加者が「一緒に編集」を開いた時、DO が Firestore から現在の `PlanData` を読み、Yjs ドキュメントを初期化。
- **②書き戻し**: DO が **部屋クローズ時 + 数分間隔の安全保存**で、Yjs ドキュメント → `PlanData` に直し、**既存の保存処理 (`toFirestoreUpdate` 相当)** で Firestore に書く。
- **誰もいない部屋は存在しない**。共同編集していない普通の表は、今まで通りただの保存済みプラン。

### 既存資産を壊さない設計原則
1. **保存・ソフトデリート (墓標)・墓標ベースマージ ([src/lib/mergePlans.ts](../../../src/lib/mergePlans.ts)) は一切変更しない。** 部屋は保存層から見れば「もう 1 つの書き手」にすぎない。
2. **共同編集は保存層の "上" に独立して乗る。** 既存コードへの変更は最小 (保存処理を部屋からも呼べるようにする程度)。
3. **既存の「コピーを配る」共有 ([src/lib/buildShareImportItems.ts](../../../src/lib/buildShareImportItems.ts)) はそのまま残す。** 用途が違う (フォーク配布 vs 同時編集)。別ボタン・別概念で共存。
4. **約束ごと**: ある表がライブ部屋で編集中のあいだ、その表の編集は部屋を通して行う (部屋 vs 部屋外の別タブが同じ表へ同時書き戻しして取り合うのを防ぐ)。

## 3. 参加権限・共有・人数上限

### 権限モデル (リンクベース / Google Docs 風)
| 論点 | 決定 |
|---|---|
| 部屋への入り方 | オーナーが「**一緒に編集リンク**」を発行 → 共有 (Discord 等) → リンクを持つ人が参加 |
| 編集の条件 | **編集はログイン必須** (Discord)。誰が編集したか分かる / presence の名前表示 / 荒らし対策 |
| 閲覧の条件 | **未ログインでも閲覧のみ可**。未ログイン接続は読み取り専用として扱い、変更メッセージを受け付けない |
| 所有権 | 表は今まで通り**オーナーのもの**。部屋はオーナーのプランへ書き戻す |
| オーナー権限 (初期) | 最低限「**セッションを終了できる**」のみ |

### 人数上限 (安全弁・厳しめスタート / 実測で調整)
- **編集者: 1 部屋 8 人** — ロックではなく「**8 つの編集席**」。8 人は全員同時編集 (CRDT で衝突なし)、9 人目以降は閲覧。**席が空けば閲覧者が編集へ昇格 / 編集者は席を譲れる** (持ち回りロックとは別物。席数だけ縛る)。
- **閲覧者: 1 部屋 20 人** — 超過したら「満員です」表示 (Liveblocks 等と同じ満員パターン)。
- **全体: 同時 30 部屋スタート** — 厳しめ。実使用量を計測して安全に引き上げる。超過したら「今混み合っています」表示。
- **緊急停止スイッチ** (共同編集を即 OFF にする feature flag)。

### 弱点と対策 (正直に明記)
リンク流出で荒らされうる。対策 = ①編集ログイン必須 (説明責任) ②オーナーがセッション終了可 ③書き戻しが既存保存処理を通るので将来「版を戻す」を後付け可能 (今は作らない)。固定パーティ用途では実用上ほぼ問題にならないと判断。

## 4. コスト設計 (最重要・正直版)

### プラン状況 (2026-06-03 確認済)
- Cloudflare アカウントは **Workers 無料プラン** (サブスクは R2 Paid $0 のみ。Workers Paid なし)。
- → **Workers / Durable Objects は無料枠超過時に "課金されずエラーで停止"** ([DO limits](https://developers.cloudflare.com/durable-objects/platform/limits/))。共同編集の心臓部に課金される仕組みは存在しない (Workers を有料に上げない限り)。

### 3 つの課金面と扱い (どこが $0 で、どこが従量か)
| 課金面 | 扱い | 共同編集の影響 |
|---|---|---|
| **Cloudflare Workers / DO** (リアルタイム処理) | 無料プラン維持 → **$0 ハードストップ** | 心臓部。$0 保証 (有料化しない限り) |
| **WebRTC TURN 中継** (カーソルの一部) | **有料 TURN は使わない**。直接 P2P 多数派は $0、厳しいネットワークの少数派 (8〜25%) はカーソル非表示 or 無料プラン DO へフォールバック | カーソルは**実質 $0 になるよう設計** |
| **Firebase / Firestore** (保存先・Google の別サービス、ほぼ確実に従量 Blaze) | **唯一の従量課金面。$0 ハードストップではない** | 保存を**部屋クローズ時 + 数分間隔の安全保存のみ**に絞り、書き込みを最小化 → 無料枠内〜ごくわずか |

### コストを生む構造 (検証済の事実)
- **送信 (配信) メッセージは課金なし / 受信プロトコル ping も課金なし** ([DO pricing](https://developers.cloudflare.com/durable-objects/platform/pricing/))。→ 編集を全員へ配信するのは無料。
- **冬眠 (Hibernation)**: アイドル DO は 10 秒で休眠し duration 課金停止、接続は維持されるが課金されない。→ 1000 部屋を開きっぱなしでも duration ≈ $0。
- 課金されるのは主に **①部屋に入る瞬間 (1 人 1 リクエスト) + ②編集の受信分 (20:1 圧縮)**。カーソルは P2P でメーター外。
- 無料枠 = **1 日 10 万リクエスト**。概算 (要負荷テスト) で **1 セッション ≈ 50〜80 リクエスト → 1 日およそ 1,000〜1,500 セッション**が無料枠内。

### 正直な結論 (誇張しない)
- ✅ 業界標準 (Yjs + 使い捨て P2P awareness + 冬眠) を真似れば、**ほぼ無料で ~1000 チーム同時接続が「圏内」**。一次資料ベースで、盛っていない。
- ⚠️ ただし「**1000 チームを恒久的に完全 $0 で保証**」とは言えない。正確な天井は**実機の負荷テストでしか確定しない**。大型実装日に延べ入室回数が数千〜数万に達すると日次予算を超え、無料プランでは「満員」($0)、必要なら上限つき少額の有料へ。
- 🛡️ **どちらに転んでも青天井にならない**: 無料プラン $0 自動停止 + 同時部屋上限 + 緊急停止 + 計測。

### 将来の昇格パス
レイド実装日に「満員で使えないのは惜しい」となったら、その時だけ Workers Paid ($5/月 + 従量) に上げ、①②③上限で従量を「事前承認した最大額」に縛る。寄付/支援が見えてからの判断。

## 5. データモデル (Yjs)

### 対象データ
`PlanData` ([src/types/index.ts:228-244](../../../src/types/index.ts#L228-L244)): `currentLevel`, `timelineEvents[]`, `timelineMitigations[]`, `phases[]`, `labels[]`, `partyMembers[]`, `aaSettings`, `schAetherflowPatterns`, `myMemberId`, `memos[]`。

### 粒度の方針
- 複数人が同時に触る心臓部 = **`timelineMitigations`** (誰がどの軽減をどこに置くか)。
- ここを **「軽減 1 個 = 1 つの Y.Map、それを Y.Array で束ねる」** 細かい粒度にする。
  - 例: A が MT に鉄壁、B が同時に ST に守りを置く → **両方残る** (取り合いで片方消えない)。
  - 雑な持ち方 (配列まるごと 1 値) だと片方が消える → **不可**。
- 同様に `timelineEvents` / `phases` / `labels` / `memos` / `partyMembers` も要素単位で Y.Map 化。
- 単一値 (`currentLevel`, `aaSettings`, `schAetherflowPatterns`, `myMemberId`) は Y.Map のエントリ = 最後に書いた人が勝ち (LWW) で十分 (同時競合しにくい項目)。

### ストアとの結合
- 共同編集セッション中は `useMitigationStore` の状態を **Yjs ドキュメントに束ねる**。ローカル編集 → Yjs 書き込み、Yjs 変更 → ストア更新 → UI 反映。
- これは「編集中に外部変更が降ってきても壊れず反映する」ための既存ストア改修を要する (本機能で一番地味に大変な箇所)。

## 6. Presence / カーソル

- **使い捨ての気配情報 = Yjs awareness** で扱う ([Yjs awareness](https://docs.yjs.dev/api/about-awareness))。保存される本体データとは別チャンネル。**P2P (WebRTC) で流し、課金サーバを通さない**。
- awareness に乗せる: `userId` (hash), 表示名, 選択ジョブアイコン, カーソル位置, 編集/閲覧の別, 編集席の状態。
- **カーソル UX (確定)**:
  - ポインターは**通常のポインターのまま**。その**右上に小さくジョブアイコン + 名前**を添える (置き換えない = 操作性維持)。
  - ログイン / 未ログイン問わずジョブアイコンを選べる。
  - 邪魔したくない閲覧者は**名前/アイコンを出さない (カーソル非配信)** を選べる → コストもかけない。
- **技術判断 (重要)**: カーソル位置は**画面ピクセル座標ではなく「タイムライン上の意味のある位置 (例: 120 秒地点・MT 行)」で送る**。理由: 利用者ごとに画面幅・拡大率・DPR が違う (本人は DPR 2.58)。ドキュメント座標で送れば、誰の画面でも正しい場所に出る。
- 編集席 (8) / 在室 / 席交代も awareness で表現。

## 7. ポリシー / 規約 (必須)

プライバシーポリシー + 利用規約 (ja/en/ko/zh の既存キー) に節を追加:
- 共同編集 (リアルタイム同時編集) の利用には**ログインが必要**。
- 参加すると**他の参加者に表示名・編集内容・在室状況がリアルタイムで見える**。
- 閲覧のみは未ログイン可だが、在室数は他者に見えうる。

## 8. 採用技術 / ライブラリ

- **Yjs** (CRDT・gold standard) + **PartyKit / y-partyserver** (Yjs を Cloudflare Durable Objects で動かす既製レイヤ) で工数圧縮 ([Cloudflare/PartyKit](https://blog.cloudflare.com/cloudflare-acquires-partykit/))。
- 編集ドキュメント = WebSocket (DO 経由・永続化と認証のため)。
- カーソル / awareness = P2P (WebRTC) で課金回避。署名サーバ (signaling) は DO か小 Worker で軽量に。
- 保存先 = 既存 Firestore (R2 は使わない)。

## 9. 実装の段取り (increments — 各段が最終形 A の一部・捨てない)

1. **DO の部屋 + WebSocket 骨組み** (PartyKit 雛形・部屋ライフサイクル)。
2. **Yjs で `PlanData` を共有型化** + `useMitigationStore` 結合 (本機能の核・一番重い)。
3. **seed (Firestore → 部屋) と書き戻し (部屋 → Firestore、debounce + クローズ時)**。
4. **presence / awareness** (P2P・ジョブアイコンカーソル・在室・編集席)。
5. **共有リンク / 参加権限 / 上限 / 緊急停止スイッチ**。
6. **ポリシー・規約の文面追加 (多言語)**。
7. **負荷テストで無料枠の実天井を計測 → 同時部屋上限を調整**。

## 10. 要検証 / 未確定 (実装計画で詰める)
- 無料枠の**実際の同時数・日次セッション上限** (負荷テスト必須・本書の数字は概算)。
- Yjs ↔ `PlanData` 双方向バインドの具体実装 (圧縮済みプラン `compressedData` との両立含む)。
- WebRTC signaling の置き場と、TURN 不使用時のフォールバック挙動 (カーソル非表示 vs DO 経由)。
- 書き戻し cadence の具体値 (Firestore 書き込み最小化と「落とさない」のバランス)。
- 編集席の昇格/譲渡 UX の詳細。
- **書き戻しの認証経路**: 有力案は **DO が信頼できるバックエンドとして Firebase Admin SDK で書く** (= client の firestore.rules をバイパスし、オーナーのプランを "サーバ代理" で更新)。これなら「オーナー以外のクライアントが他人のプランを書く」ための rules 緩和が不要で、既存のオーナー書き込み経路・墓標・楽観ロックを壊さない。Admin SDK 資格情報の DO への安全な持たせ方 (Secret) は実装計画で確定。代替案 (各自が自分の権限で書く) は rules を緩める必要があり既存設計と衝突するため非推奨。
