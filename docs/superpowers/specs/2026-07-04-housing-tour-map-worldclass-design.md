## 0. 目的

408770d1 まででツアー中ページの「箱ハイライト・アパートナビ・座標」は正しくなった。本 spec は、そこから中央地図とツアー進行を **本物のカーナビ水準**へ引き上げる。合言葉は「開発者が見ている画面 = 世界に届けたい UI」。

10 改善を 4 グループに束ねる:

- **A. 地図のナビ描画**(改善 1・2・3・8)
- **B. 地図の操作と余白**(改善 4・5・6・7)
- **C. レイアウト再編**(改善 9)
- **D. ツアー進行モデルの拡張**(改善 10)

## 1. 全体像 — 3 カラムの新しい役割

現状(`TourNavPage.tsx`)は 左=進捗パネル / 中央=地図 / 右=次の目的地(写真+詳細+ステップ+操作+報告)で、右が「全部乗せ」になっている。これを役割で 3 分割する:

| カラム | 新しい役割 | 主な中身 |
|---|---|---|
| **左** | 目的地ショーケース(この家の魅力) | 大きな写真(将来スライドショー/動画)+ 住所/サイズ/ワールド/最寄りエーテライト/ひとことメモ + 操作(前へ/見学を始める/次へ)+ 報告 |
| **中央** | ナビ(どう行くか) | 地図。エーテライト起点 → 家までの経路 + 光る箱。パン&ズーム可 |
| **右** | ツアー進行状況(どこまで進んだか) | 進捗(リング or バー)+ 全ステップの縦リスト(移動節目込み・青い線が進む)+ 見学タイマー |

## 2. グループ A — 地図のナビ描画

### 改善 1: 出発を「エーテライトのすぐ前の道」から始める

- **現状(事実)**: 起点マーカー(`origin`)はエーテライトの実座標に描く(`buildTourMapPlacements.ts:65`)。しかし経路は `buildRoutePathIn(json, originInfo.node, …)` で **エーテライトに紐づく最寄りノード(`shard.node`)から** 始まる(同 `:63`、`getPlotOriginNode` = `plotOrigin.ts:21-23` / `getApartmentOrigin` = `apartmentOrigin.ts`)。エーテライト座標にノードは無いため、**マーカーと経路始点が乖離**し「道が繋がって見えない」。
- **変更**: エーテライト座標を、全 edge(`json.edges[].polyline`)の全セグメントへ投影し、**幾何的に最も近い「道の上の点」**(線分への垂線の足)を求める。そこを経路の視覚的始点にする(= カーナビの現在地スナップ)。
- **見せ方**: エーテライトの光る丸は実座標のまま。丸 →(投影点)→ 経路、と繋ぐ。丸を道の上に寄せる/寄せないの微差は実装後に実画面で最終調整(ブレストで「今決めない」合意)。
- **アプローチ**: 点→線分の最近点は純関数(`nearestPointOnPolylines(origin, edges)` を `wardRoute.ts` 系に追加)。既存の座標正規化(0..1 × viewBox)に一致させる。

### 改善 2: 到着は「家の手前」で止める(通り過ぎ・突き刺し解消)

- **現状(事実)**: house は箱の**中心** `x,y` と 紐づく `node` だけを持ち、**玄関(入口)座標は持たない**(`wardMapManifest.ts:6`)。`house.node` は「箱中心に幾何的に最も近いノード」(`parse-ward-svg.mjs:130` の `nearestNode`)で、家のどちら側かは考慮されない。経路は最後に `L 箱中心`(`buildTourMapPlacements.ts:64`)で終わるため、線が箱の中心まで突き刺さり、ノードが箱の裏/奥だと回り込み・通り過ぎに見える。
- **変更(軽め案で確定)**: 経路の最後のセグメント(箱へ近づく向き)が **箱の輪郭と交わる点**で経路を止める。箱の中には入らない。新規データ不要 — 箱の輪郭頂点は `svg-path-center.mjs` の `pathPoints(d)`(+ `applyTransform` で rotate 対応)で取得できる。
- **玄関の向き**: 「箱の手前で止まる(通り過ぎ・突き刺し解消)」を確実にやる。FF14 の実際の玄関向きに合わせる"こだわり案"は、実機で見て気になったときに格上げ(全 310 軒の玄関辺算出 + 実機妥当性チェック)。今回スコープは軽め案。
- **アプローチ**: `segmentBoxIntersection(lastSeg, boxPolygon)` 純関数。箱要素(`#plot_N` / `#apart_N`)の輪郭は SVG から取得。ビルド時(generated.json に door 点を焼く)/実行時(SVG パース)のどちらで持つかは実装フェーズで決定(改善 1 と同じ幾何基盤に載せる)。

### 改善 3: 家から出る「波紋」を撤去

- **現状(事実)**: 目的地の放射リング波紋 = `TourNavMap.tsx:73-82`(`target` ブロック、r 60→170 の circle × 2)。
- **変更**: この放射リングを削除。**エーテライトの脈動**(`:63-71` origin pulse)と **箱本体の光るハイライト**(`:25-32` の `.housing-tour-target-box` useEffect)は**維持**。目的地は「光る箱」で示す。

### 改善 8: 地図の黒枠を世界観の色へ

- **現状(事実)**: 生成 SVG の全要素が **純黒の輪郭**。道全体 = `stroke="black" stroke-width="8"`、各家/アパート = `stroke="black" stroke-width="4"`(`mist.generated.svg` ほか 10 マップ)。塗りは `fill="white" fill-opacity="0.3"`。これは Figma 書き出しの素のままで、意匠選択ではない。
- **評価**: 純黒はハウジングの「濃紺ガラス + 動画背景 + ハニーゴールドの灯火」の世界観に対して硬く製図的で浮く。
- **変更**: **輪郭は残す**(消すと区画境界が読めなくなる)。純黒 → 世界観に馴染む色へ。候補は (a) 白の半透明(暗背景に浮かぶ光の線=ミニマップ感) / (b) ハニー/キャンドル系(魔法の地図感)。
- **アプローチ**: SVG の `stroke` は presentation attribute なので **CSS が勝つ**。SVG を書き換えず `.housing-map-svg-host` 配下の道・区画パスへ CSS で `stroke` を上書きできる(装飾要素=エーテライト Polygon `#F0FEFF` / `#00BFFF` などに波及しないようセレクタを限定)。`stroke-width` の差(道 8 / 箱 4)は保持。昼/夜テーマ連動も可能。**具体色・濃さは実画面ゲートで 2〜3 案を見比べて確定**(トークン化して housing.css に定義)。

## 3. グループ B — 地図の操作と余白

### 改善 4: 地図上の番号マーカー(①②③・✓)を撤去

- **現状(事実)**: `TourNavMap.tsx:84-88` の `model.placed` を番号/✓ の div で描画。
- **変更**: これを削除。目的地は「光る箱」で示す(同一地図に複数ステップがあっても、順番は右パネルのステップ一覧で分かる)。

### 改善 6: 説明 UI(LIVE + 凡例)を撤去

- **現状(事実)**: 左上 LIVE ピル = `TourNavMap.tsx:92`、テキストは `housing.tour.nav.live`(ja「情報は自動で更新されます」= 開発者向け内部説明、`ja.json:2752` ほか 4 言語)。下部凡例 = `:94-96`(here/next/arrived/upcoming/route の 5 項目)。
- **変更**: LIVE ピルを**緑ドットごと丸ごと削除**。下部凡例も**全削除**。ナビは見れば分かる(カーナビに凡例は無い)。i18n の `live` / `legend.*` キーは死にキー化(削除 or 保留は実装時判断)。
- **統合**: 改善 4 と本項は「地図から番号と凡例を撤去」で 1 タスクに束ねる。

### 改善 5 + 7: パン&ズーム と デフォルト表示(パネル充填)

改善 5(操作の追加)と改善 7(デフォルトの見え方)は同じ「地図ビューポート」機構なので 1 つに束ねる。

- **現状(事実)**: 地図の入れ物 `.housing-tour-map-wrap` は `width: 88%` + `aspect-ratio: 470/350`(横長固定)で stage 中央に絶対配置(`housing.css:5994-6001`)。中央パネルは縦長なので、横長地図が中央に小さく浮き、**左右 12% + 上下に大きな余白**(= すかすか)。地図・経路・エーテライト・箱ハイライトは**同一座標系**で重なっている(host + overlay + useEffect ハイライト)。
- **パン&ズーム(改善 5)**:
  - 地図中身(`.housing-tour-map-wrap` 相当)を **1 つの入れ物ごと** `transform: translate() scale()` で動かす → 全レイヤーがズレずに一緒に動く。HUD/操作ボタンは外側=固定。
  - ドラッグでパン(縦横斜め自由)、ホイールでズーム(**カーソル位置中心**)。
  - **「全体に戻す」ボタン**を置く(必須)。
  - **スマホ対応必須**: ピンチズーム + 1 本指ドラッグ(pointer events で PC/タッチ統一)。
  - **実装方式(ライブラリ vs 自前)は実装フェーズで事実ベースに調査して最良選択**(依存追加の是非込み。ユーザー委任)。候補: `react-zoom-pan-pinch` 等の軽量ライブラリ / pointer events 自前。
- **デフォルト表示(改善 7)**:
  - デフォルト = **今の経路(エーテライト 〜 家)にフィットしてパネルの縦横を埋める**。地図全体を meet で小さく出すのをやめる。
  - **絶対制約**: 最寄りエーテライトと向かう家が **上下左右どこにも見切れない**こと(必ず両方が可視領域に収まるよう、経路 bbox にパディングを足し、パネル比に合わせて拡張、地図範囲内でクランプ)。
  - **次のステップに進んだら、このデフォルト表示に自動リセット**(改善 5 で確定)。手動パン/ズームはその後ユーザーが自由に。
- **アプローチ**: 「経路 bbox → パネル比に合わせた表示 rect → transform」を純関数化(`computeDefaultView(originPt, housePt, panelAspect, mapViewBox)`)。見切れ厳禁はここでテストする。

## 5. グループ D — ツアー進行モデルの拡張(改善 10)

現状は `useHousingTourStore`(`currentIndex` + `next`/`prev`)+ `tourNav.ts`(`resolveTourSteps` / `computeTourProgress`)で「家のステップ」を線形に進むだけ。これを 3 点で拡張する。

### D-1: 移動の節目をステップに織り込む

- **データ**: 各 listing は `dc` / `server`(ワールド)/ `area` / `ward` を持つ(`mockListings.ts:16-20`)。連続 2 軒のこの 4 つを比較して移動節目を導出。
- **判定(FF14 移動階層 — 実装前にユーザー最終確認)**:

  | 2 軒の関係 | 挿入する節目 |
  |---|---|
  | 別 `dc` | **DC トラベル** |
  | 同 `dc`・別 `server` | **ワールド移動(ワールド訪問)** |
  | 同 `server`・別 `area` or 別 `ward` | **テレポ(最寄りエーテライト)** |
  | 同 `area` 同 `ward` | **徒歩**(= 中央地図のナビ) |

  上位が違えば上位から順に複数節目が入る(例: 別 DC → DC トラベル → ワールド移動 → テレポ → 徒歩)。
- **表示**: 右パネルの縦リストが「家(見学済)→ DC トラベル → ワールド移動 → テレポ → 家(次)」のように、やるべき移動が全部見える。
- **アプローチ**: `resolveTourSteps` の出力を、家ステップ + 移動節目ステップの列に拡張する純関数(`buildTourItinerary(steps)` を新設。既存 `TourStep` は壊さず、`TourItineraryItem = HouseStep | TransitionStep` の派生列を作る)。

### D-2: 「移動(次へ)」と「見学を始める」を分ける

- **現状**: 「到着した→次へ」の 1 ボタン(`TourNavPage.tsx` `onPrimary` → `next()`)。
- **変更**: ツアーを **移動フェーズ ↔ 見学フェーズ** の 2 状態にする。フロー:

  ```
  [移動中] 節目を「次へ」で進める(Enter / Space / Shift 等のキー操作対応。ツアー中の連続操作向け)
     ↓ 家に到着
  [到着]  「見学を始める」ボタンが出る
     ↓ 押す
  [見学中] 見学タイマー作動(D-3)
     ↓ 「次へ」で次の移動へ
  ```
- **キー操作**: 既存要件メモ(`docs/housing-tour-planner-requirements.md`)にも「PC は Enter/Space で次へ」とあり方向は一致。各キーの割り当て(次へ / 見学を始める にどのキーを当てるか)は実装フェーズで確定。
- **状態管理**: `useHousingTourStore` に phase(`moving` | `viewing`)を追加(既存 `currentIndex` は非破壊)。

### D-3: 見学タイマー

- **変更**: 「見学を始める」を押すと、**見学開始時刻**を記録し、**経過時間**をカウント。右パネル(進行状況)に表示(例:「見学開始 14:32 / 経過 3 分」)。
- **動機**: 大所帯ツアーで「1 軒 ○ 分」と区切って全員のペースを合わせる。一人のときは無視すればよく邪魔にならない。
- **見せ方(常時 or 折りたたみ)・記録の永続化有無**は実装フェーズで詰める(M1 は永続化なしのセッション内で十分)。

## 6. 実装フェーズ(順序 + モデル割り当て方針)

各フェーズ末にユーザー実機ゲート。モデルは既存ツアー開発と同じ布陣(純関数/i18n/座標計算 = haiku、UI/統合 = sonnet、最終レビュー = 最上位)。

   - 幾何純関数(点→道投影 / 経路×箱交点 / 枠線色トークン)= haiku
   - 波紋撤去 + 枠線色 CSS = sonnet、色は実画面ゲート
2. **Phase 2 — 地図の撤去 + ビューポート**(改善 4・6・5・7)
   - 番号/LIVE/凡例 撤去 = haiku 寄り
   - `computeDefaultView`(見切れ厳禁テスト)= haiku、パン&ズーム機構 = sonnet(方式は事前調査)
3. **Phase 3 — レイアウト再編**(改善 9・写真枠は静止で用意)= sonnet
4. **Phase 4 — 進行モデル拡張**(改善 10 = 移動節目 / ボタン分離 / タイマー)
   - `buildTourItinerary` 純関数 + FF14 判定 = haiku(ユーザー確認後)
   - store phase + UI + キー操作 = sonnet
5. **Phase 5 — 生きたカード段階 2**(改善 9 の写真 = スライドショー/動画)= `HousingPlaybackProvider` 配線 + Allmarks 流用。探すページ含む横断機能。ユーザー指示で本 spec スコープに含む(2026-07-04)。spotlight 再生 + 複数画像スライドショー。

各フェーズは前フェーズの実機 OK を待って着手。最終ブランチレビュー(最上位)→ 実機ゲート → merge。

## 9. テスト方針

- 幾何純関数(投影 / 交点 / デフォルト表示 rect)は happy-dom 不要の純ユニットテスト。**改善 7 の「エーテライトと家が見切れない」は明示的にテスト**(経路 bbox が表示 rect に必ず内包される)。
- FF14 移動判定(`buildTourItinerary`)は代表 4 パターン(同 ward / 別 area / 別 world / 別 dc)+ 多段(別 dc → 複数節目)をテスト。
- i18n は 4 言語 parity(新規キー = 移動節目名 / 見学タイマー / 操作ボタン)。
- `npm run build`(tsc -b 厳密)EXIT0 + 既知 legacy 5 fail 以外の新規 fail ゼロ。
- 見た目(枠線色 / デフォルト表示 / レイアウト)は実画面 CSS1489・DPR2.58 ゲート。
